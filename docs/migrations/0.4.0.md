# Migrating from Autumn `0.3` to `0.4`

## At a glance

- **Old version:** `autumn-web 0.3.x`
- **New version:** `autumn-web 0.4.0`
- **Expected upgrade effort:** small for local-only apps; medium for production
  apps using sessions, signed storage URLs, repository-generated APIs, mail
  `deliver_later`, or the removed `storage-s3` feature.
- **MSRV delta:** `1.88.0` -> `1.88.0` (unchanged)
- **Carried dependency majors:** none

## Summary

Autumn 0.4.0 tightens production defaults around signed state, generated APIs,
durable mail delivery, and storage plugins. Most source-compatible apps only
need a dependency bump, but production apps should review the configuration
changes below before rolling the new binary.

## Before you start

1. Pin your current dependency (`autumn-web = "=0.3.0"`) and commit.
2. Make sure your tests are green on 0.3.x.
3. Upgrade the Autumn crates together:

   ```toml
   [dependencies]
   autumn-web = "0.4"
   ```

## Breaking changes

### Security: production signing secret is required

Production profiles now refuse to bind unless a stable signing secret is
configured. This protects sessions, CSRF tokens, flash/signed-cookie state, and
signed storage URLs from per-process key churn.

Set a generated value before starting a production process:

```bash
export AUTUMN_SECURITY__SIGNING_SECRET="$(openssl rand -hex 32)"
```

For rotation, configure the new secret as `secret` and keep the old value in
`previous_secrets` under `security.signing_secret` until existing cookies and
tokens have expired:

```toml
[security.signing_secret]
previous_secrets = ["old-secret-hex-value"]
```

### Storage: `storage-s3` moved to a plugin crate

The `storage-s3` feature no longer exists on `autumn-web`.

```toml
# 0.3.x
autumn-web = { version = "0.3", features = ["storage", "storage-s3"] }

# 0.4.0
autumn-web        = { version = "0.4", features = ["storage"] }
autumn-storage-s3 = "0.4"
```

Then install the S3 store through the app builder:

```rust,ignore
let store = S3BlobStore::from_config(&config.storage.s3).await?;

autumn_web::app()
    .with_blob_store(store)
    .run()
    .await;
```

### Authorization: repository APIs require a policy in production

Generated repository APIs must be paired with a policy in `prod`/`production`.

```rust,ignore
#[repository(Post, api = "/api/posts", policy = PostPolicy, scope = PostScope)]
pub trait PostRepository {}
```

If a generated API is intentionally unauthenticated, acknowledge that in
configuration:

```toml
[security]
allow_unauthorized_repository_api = true
```

### Mail: production `deliver_later` must be durable or acknowledged

`Mailer::deliver_later` and generated `deliver_later_*` helpers now require a
durable `MailDeliveryQueue` in production unless mail transport is disabled.
Register the queue before `.run()`:

```rust,ignore
autumn_web::app()
    .with_mail_delivery_queue(queue)
    .run()
    .await;
```

For a deliberate non-durable production fallback, set:

```toml
[mail]
allow_in_process_deliver_later_in_production = true
```

## Configuration changes

- `AUTUMN_SECURITY__SIGNING_SECRET` is required in `prod` / `production`.
- `[security.signing_secret] previous_secrets = [...]` supports signing-secret
  rotation.
- `[security] allow_unauthorized_repository_api = true` is the explicit escape
  hatch for production repository APIs without a policy.
- `[mail] allow_in_process_deliver_later_in_production = true` is the explicit
  escape hatch for non-durable deferred mail in production.

## Troubleshooting

- Run `autumn doctor --strict` before deployment. It includes the
  `signing_secret` check and reports configuration failures without printing
  secret values.
- If Cargo reports `storage-s3` is not a feature of `autumn-web`, remove it
  from the `autumn-web` dependency and add `autumn-storage-s3 = "0.4"`.
- If production startup fails for a generated repository API, add `policy = ...`
  or set `allow_unauthorized_repository_api = true` intentionally.
