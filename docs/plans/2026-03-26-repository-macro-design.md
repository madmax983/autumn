# `#[repository]` Macro Design

**Date:** 2026-03-26
**Status:** Validated (post six-hats review)
**Target:** v0.2.0

## Overview

A proc macro that generates async CRUD operations and derived queries for Diesel models, exposed as an Axum extractor. Inspired by Spring Data JPA — compile-time codegen instead of runtime reflection.

**Differentiators vs Spring Data:**
- Compile-time query validation (not runtime startup)
- Compile-time missing index warnings (no framework in any language does this)
- Type-safe insert/update split (not runtime ID check)
- Trait-based mockability for testing

## User-Facing API

```rust
// 1. Define model (generates query, insert, and update types)
#[model]
pub struct Post {
    #[id]
    id: i32,
    #[indexed]
    title: String,
    body: String,
    #[indexed]
    published: bool,
}
// Generates: Post (Queryable), NewPost (Insertable), UpdatePost (AsChangeset)
// Generates: index migration hints for title, published

// 2. Declare repository with derived queries
#[repository(Post)]
trait PostRepository {
    // Auto-generated CRUD (always present, no declaration needed):
    //   find_by_id(id: i32) -> Option<Post>
    //   find_all() -> Vec<Post>
    //   save(new: &NewPost) -> Post
    //   update(id: i32, changes: &UpdatePost) -> Post
    //   delete_by_id(id: i32) -> ()
    //   count() -> i64
    //   exists_by_id(id: i32) -> bool

    // Custom derived queries:
    fn find_by_published(published: bool) -> Vec<Post>;
    fn find_by_title_and_published(title: &str, published: bool) -> Option<Post>;
    fn count_by_published(published: bool) -> i64;
    fn delete_by_published(published: bool) -> ();
}

// 3. Use as extractor
#[get("/posts")]
async fn list(repo: PostRepository) -> AutumnResult<Json<Vec<Post>>> {
    Ok(Json(repo.find_by_published(true).await?))
}

// 4. Custom queries alongside generated ones
impl PostRepository {
    pub async fn find_recent_published(&self, limit: i64) -> AutumnResult<Vec<Post>> {
        let conn = self.pool.get().await?;
        conn.interact(move |conn| {
            posts::table
                .filter(posts::published.eq(true))
                .order(posts::id.desc())
                .limit(limit)
                .load::<Post>(conn)
        }).await?
    }
}
```

## Design Decisions

### Trait stays alive (mockability)
The macro generates both a trait and a concrete struct. The user-declared trait becomes the public interface. A `PgPostRepository` struct implements it. This preserves mockability for testing:

```rust
// Generated by macro:
pub trait PostRepository {
    async fn find_by_id(&self, id: i32) -> AutumnResult<Option<Post>>;
    async fn find_all(&self) -> AutumnResult<Vec<Post>>;
    async fn save(&self, new: &NewPost) -> AutumnResult<Post>;
    // ... all CRUD + derived queries
}

pub struct PgPostRepository {
    pool: deadpool_diesel::postgres::Pool,
}

impl PostRepository for PgPostRepository { ... }

// User can mock in tests:
struct MockPostRepository;
impl PostRepository for MockPostRepository { ... }
```

The extractor returns `PgPostRepository` but handlers can accept `impl PostRepository` for testability.

### Explicit return types, implicit error wrapping
User writes the happy path type (`Vec<Post>`, `Option<Post>`). The macro wraps it in `AutumnResult<T>` and adds `async` + `&self`. Reduces noise without hiding what the query returns.

### Return type semantics
- `Option<T>` -> `.first().optional()` (single result or none)
- `Vec<T>` -> `.load()` (multiple results)
- `()` -> `.execute()` (for deletes)
- `i64` -> `.count().get_result()` (for counts)
- `bool` -> count check > 0 (for exists)

### `Option<T>` field semantics
When a model field is `Option<String>` (e.g., `bio: Option<String>`):
- `find_by_bio(bio: &str)` -> filters `WHERE bio = $1` (matches non-null values)
- `find_by_bio(bio: Option<&str>)` -> `None` means `WHERE bio IS NULL`, `Some(v)` means `WHERE bio = v`
- The parameter type in the trait signature determines the behavior

### Repository-as-extractor
Implements `FromRequestParts`, pulls the connection pool from `AppState`. Consistent with existing `Db` extractor pattern. Zero boilerplate in handlers.

### Table name inference
Same convention as `#[model]`: `Post` -> `posts` (PascalCase -> snake_case + 's'). Override with `#[repository(Post, table = "blog_posts")]`.

### Insert/Update split (no unified `save()`)
- `save(&NewPost) -> Post` for inserts
- `update(id, &UpdatePost) -> Post` for updates

Rust's type system enforces this at compile time. `NewPost` has no `id` field — you can't accidentally update when you meant to insert. Safer than Spring's runtime ID check.

### Custom query escape hatch
Users can add their own `impl PgPostRepository` block alongside the generated one. Rust allows multiple impl blocks on the same struct. This is the escape hatch for queries too complex for derivation.

## `#[model]` Enhancement

The `#[model]` macro generates three types (breaking change for v0.2.0):

```rust
#[model]
struct Post {
    #[id]
    id: i32,
    #[indexed]
    title: String,
    body: String,
    #[indexed]
    published: bool,
}
```

Generates:

```rust
#[derive(Queryable, Selectable, Serialize, Deserialize)]
#[diesel(table_name = posts)]
pub struct Post {
    pub id: i32,
    pub title: String,
    pub body: String,
    pub published: bool,
}

#[derive(Insertable, Serialize, Deserialize)]
#[diesel(table_name = posts)]
pub struct NewPost {
    pub title: String,
    pub body: String,
    pub published: bool,
}

#[derive(AsChangeset, Serialize, Deserialize)]
#[diesel(table_name = posts)]
pub struct UpdatePost {
    pub title: Option<String>,
    pub body: Option<String>,
    pub published: Option<bool>,
}
```

### Primary key detection
- Default: first `i32` or `i64` field
- Explicit: `#[id]` attribute on one or more fields
- The ID field(s) are excluded from `NewPost` and `UpdatePost`

### Composite primary keys
Multiple `#[id]` fields are supported:

```rust
#[model]
struct TenantPost {
    #[id]
    tenant_id: i32,
    #[id]
    post_id: i32,
    title: String,
}
```

Generated CRUD adapts:
- `find_by_id(tenant_id: i32, post_id: i32) -> Option<TenantPost>`
- `delete_by_id(tenant_id: i32, post_id: i32) -> ()`
- `update(tenant_id: i32, post_id: i32, changes: &UpdateTenantPost) -> TenantPost`
- `NewTenantPost` excludes both `tenant_id` and `post_id`

### `#[indexed]` attribute
Marks fields that should have database indexes. Three compile-time features:

**1. Compile-time warning on unindexed queries:**
When `#[repository]` generates a `find_by_*` query on a field not marked `#[indexed]`, it emits:
```
warning: `find_by_category` queries field `category` which is not marked #[indexed].
         Consider adding #[indexed] to `Post::category` or adding a database index.
```

**2. Migration hint generation:**
The `#[model]` macro generates a companion function with suggested SQL:
```rust
impl Post {
    /// Suggested index migrations based on #[indexed] fields:
    /// ```sql
    /// CREATE INDEX idx_posts_title ON posts (title);
    /// CREATE INDEX idx_posts_published ON posts (published);
    /// ```
    pub fn index_hints() -> &'static str { ... }
}
```

**3. Cross-reference validation:**
When `#[repository]` sees `find_by_title_and_published`, it checks:
- `title` is `#[indexed]` ✓
- `published` is `#[indexed]` ✓
- Compound query → suggests composite index:
```
hint: `find_by_title_and_published` queries multiple fields.
      Consider a composite index: CREATE INDEX idx_posts_title_published ON posts (title, published);
```

## Query Name Parser

### Grammar (Tier 1)
```
method    := prefix "_by_" field_list
prefix    := "find" | "count" | "delete" | "exists"
field_list := field (("_and_" | "_or_") field)*
field     := <model field name in snake_case>
```

### Constraint: no mixing `and`/`or`
`find_by_a_and_b_or_c` is rejected at compile time. Ambiguous precedence. User drops to raw Diesel for complex predicates.

### Compile error messages
- Unknown field: `"Unknown field 'titl' on model 'Post'. Did you mean 'title'?"`
- Type mismatch: `"Parameter type mismatch: field 'title' is String, got bool"`
- Trailing combinator: `"Expected field name after 'and'"`
- Mixed combinators: `"Cannot mix 'and' and 'or' in a single query — use a custom Diesel query instead"`

### Designed for Tier 2 extension
The parser splits on keywords in order. Adding `_greater_than`, `_containing`, `_is_null`, `_starts_with` later means inserting new keyword checks between field name resolution and parameter validation. No structural changes needed.

**Tier 2 keyword reservation:** Field names that collide with future operator suffixes (`greater`, `than`, `containing`, `starts`, `ends`, `null`, `not`) will emit a warning at Tier 1 suggesting explicit `#[field_name = "..."]` disambiguation if they plan to use Tier 2 features later. This is a warning, not an error — Tier 1 has no operators so there's no ambiguity yet.

## Generated Code

### Trait + concrete struct
```rust
// The trait (public interface, mockable)
pub trait PostRepository: Send + Sync {
    async fn find_by_id(&self, id: i32) -> AutumnResult<Option<Post>>;
    async fn find_all(&self) -> AutumnResult<Vec<Post>>;
    async fn save(&self, new: &NewPost) -> AutumnResult<Post>;
    async fn update(&self, id: i32, changes: &UpdatePost) -> AutumnResult<Post>;
    async fn delete_by_id(&self, id: i32) -> AutumnResult<()>;
    async fn count(&self) -> AutumnResult<i64>;
    async fn exists_by_id(&self, id: i32) -> AutumnResult<bool>;
    // + derived queries from trait body
    async fn find_by_published(&self, published: bool) -> AutumnResult<Vec<Post>>;
    async fn find_by_title_and_published(&self, title: &str, published: bool) -> AutumnResult<Option<Post>>;
    async fn count_by_published(&self, published: bool) -> AutumnResult<i64>;
    async fn delete_by_published(&self, published: bool) -> AutumnResult<()>;
}

// The concrete Postgres implementation
pub struct PgPostRepository {
    pool: deadpool_diesel::postgres::Pool,
}
```

### CRUD impl
```rust
impl PostRepository for PgPostRepository {
    async fn find_by_id(&self, id: i32) -> AutumnResult<Option<Post>> {
        let conn = self.pool.get().await?;
        conn.interact(|conn| {
            posts::table.find(id).first::<Post>(conn).optional()
        }).await?
    }

    async fn find_all(&self) -> AutumnResult<Vec<Post>> {
        let conn = self.pool.get().await?;
        conn.interact(|conn| {
            posts::table.load::<Post>(conn)
        }).await?
    }

    async fn save(&self, new: &NewPost) -> AutumnResult<Post> {
        let conn = self.pool.get().await?;
        conn.interact(|conn| {
            diesel::insert_into(posts::table)
                .values(new)
                .get_result::<Post>(conn)
        }).await?
    }

    async fn update(&self, id: i32, changes: &UpdatePost) -> AutumnResult<Post> {
        let conn = self.pool.get().await?;
        conn.interact(|conn| {
            diesel::update(posts::table.find(id))
                .set(changes)
                .get_result::<Post>(conn)
        }).await?
    }

    async fn delete_by_id(&self, id: i32) -> AutumnResult<()> {
        let conn = self.pool.get().await?;
        conn.interact(|conn| {
            diesel::delete(posts::table.find(id)).execute(conn)
        }).await?;
        Ok(())
    }

    async fn count(&self) -> AutumnResult<i64> {
        let conn = self.pool.get().await?;
        conn.interact(|conn| {
            posts::table.count().get_result::<i64>(conn)
        }).await?
    }

    async fn exists_by_id(&self, id: i32) -> AutumnResult<bool> {
        let conn = self.pool.get().await?;
        conn.interact(|conn| {
            diesel::select(diesel::dsl::exists(posts::table.find(id)))
                .get_result::<bool>(conn)
        }).await?
    }
}
```

### Derived query impl (example)
```rust
impl PostRepository for PgPostRepository {
    async fn find_by_title(&self, title: &str) -> AutumnResult<Vec<Post>> {
        let title = title.to_owned();
        let conn = self.pool.get().await?;
        conn.interact(move |conn| {
            posts::table
                .filter(posts::title.eq(&title))
                .load::<Post>(conn)
        }).await?
    }

    async fn find_by_title_and_published(
        &self, title: &str, published: bool
    ) -> AutumnResult<Option<Post>> {
        let title = title.to_owned();
        let conn = self.pool.get().await?;
        conn.interact(move |conn| {
            posts::table
                .filter(posts::title.eq(&title))
                .filter(posts::published.eq(published))
                .first::<Post>(conn)
                .optional()
        }).await?
    }
}
```

### Extractor impl
```rust
#[axum::async_trait]
impl<S> FromRequestParts<S> for PgPostRepository
where
    S: Send + Sync,
    AppState: FromRef<S>,
{
    type Rejection = AutumnError;

    async fn from_request_parts(
        _parts: &mut Parts,
        state: &S,
    ) -> Result<Self, Self::Rejection> {
        let app_state = AppState::from_ref(state);
        let pool = app_state.pool()
            .ok_or_else(|| AutumnError::internal("No database pool configured"))?
            .clone();
        Ok(PgPostRepository { pool })
    }
}
```

## Full Handler Example

```rust
use autumn_web::{get, post, put, delete, Json, extract::Path, AutumnResult, AutumnError};
use crate::models::{Post, NewPost, UpdatePost};
use crate::repositories::PostRepository;

#[get("/posts")]
async fn list(repo: PgPostRepository) -> AutumnResult<Json<Vec<Post>>> {
    Ok(Json(repo.find_by_published(true).await?))
}

#[get("/posts/{id}")]
async fn show(repo: PgPostRepository, id: Path<i32>) -> AutumnResult<Json<Post>> {
    repo.find_by_id(*id).await?
        .map(Json)
        .ok_or_else(|| AutumnError::not_found("Post not found"))
}

#[post("/posts")]
async fn create(repo: PgPostRepository, Json(new): Json<NewPost>) -> AutumnResult<Json<Post>> {
    Ok(Json(repo.save(&new).await?))
}

#[put("/posts/{id}")]
async fn update(
    repo: PgPostRepository, id: Path<i32>, Json(changes): Json<UpdatePost>
) -> AutumnResult<Json<Post>> {
    Ok(Json(repo.update(*id, &changes).await?))
}

#[delete("/posts/{id}")]
async fn destroy(repo: PgPostRepository, id: Path<i32>) -> AutumnResult<()> {
    repo.delete_by_id(*id).await
}
```

## Spring Boot Comparison

| Spring Boot | Autumn |
|---|---|
| `@Entity Post` | `#[model] struct Post` |
| `@Id`, `@GeneratedValue` | `#[id]` |
| `@Index(columnList = "title")` | `#[indexed]` (+ compile-time warnings) |
| `interface PostRepository extends JpaRepository<Post, Long>` | `#[repository(Post)] trait PostRepository` |
| `@Autowired PostRepository repo` | `repo: PgPostRepository` (extractor) |
| `repo.findByPublished(true)` | `repo.find_by_published(true).await?` |
| Runtime reflection + proxy | Compile-time codegen + type checking |
| `CrudRepository` (6 methods) | 7 auto-generated methods |
| `findByTitleAndPublished` | `find_by_title_and_published` |
| Runtime exceptions | `AutumnResult<T>` compile-time |
| No index awareness | Compile-time missing index warnings |
| `@Table(indexes = ...)` only documents | `#[indexed]` validates + suggests migrations |
| Composite keys via `@EmbeddedId` | Multiple `#[id]` fields |
| Mock via `@MockBean` | Implement the trait directly |

## Implementation Order

1. **Enhance `#[model]`** — Generate `NewPost` + `UpdatePost` types, `#[id]` attribute, `#[indexed]` attribute with migration hints
2. **Query name parser** — Standalone module: parse method names, validate against field list, error messages with suggestions
3. **`#[repository]` codegen** — Generate trait + `PgXxxRepository` struct + CRUD impl + derived query impl
4. **Index cross-reference** — Warn when derived queries target unindexed fields, suggest composite indexes
5. **Extractor impl** — `FromRequestParts` for repository struct
6. **Re-exports** — `autumn_web::repository` macro, generated types accessible from prelude
7. **Tests** — Unit tests for parser, trybuild compile-fail tests, mock trait tests, integration test with real DB
8. **Example** — Update todo-app or blog example to use repositories

## Risks & Mitigations (from Six Hats Review)

| Risk | Mitigation |
|---|---|
| Diesel trait bound errors in generated code are hard to debug | Generate `#[doc(hidden)]` helper that users can `cargo expand` to see generated code. Add custom error messages where possible. |
| `interact()` closures require owned data, hiding allocations | Document the ownership cost. For hot paths, users drop to custom impl with pre-allocated buffers. |
| Diesel schema module must exist for generated code to compile | Emit clear error: "Repository `PostRepository` references table `posts`. Ensure `diesel::table! { posts { ... } }` exists in your schema." |
| Field names colliding with Tier 2 operator keywords | Emit warning at Tier 1 for reserved-looking field names. No ambiguity at Tier 1, but early warning for future. |
| Diesel major version bump breaks generated code | Pin diesel version in autumn-web. Generated code uses only stable Diesel APIs (filter, eq, load, insert_into, etc.). |

## Tier 2 Roadmap (Future)

Operators to add when Tier 1 is stable:

| Suffix | SQL | Diesel |
|---|---|---|
| `_greater_than` | `> $1` | `.gt()` |
| `_less_than` | `< $1` | `.lt()` |
| `_greater_than_equal` | `>= $1` | `.ge()` |
| `_less_than_equal` | `<= $1` | `.le()` |
| `_containing` | `LIKE '%$1%'` | `.like(format!("%{val}%"))` |
| `_starts_with` | `LIKE '$1%'` | `.like(format!("{val}%"))` |
| `_ends_with` | `LIKE '%$1'` | `.like(format!("%{val}"))` |
| `_is_null` | `IS NULL` | `.is_null()` (no param) |
| `_is_not_null` | `IS NOT NULL` | `.is_not_null()` (no param) |
| `_in` | `IN ($1)` | `.eq_any()` |
| `_not_in` | `NOT IN ($1)` | `.ne_all()` |

## Tier 3 Roadmap (Future)

- `_order_by_{field}_asc` / `_desc` suffixes
- `find_first_by_*` (LIMIT 1)
- Pagination: `find_by_published(published: bool, page: Page) -> PageResult<Post>`
