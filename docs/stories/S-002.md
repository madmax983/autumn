## User Story

As a **framework user**
I want to **annotate an async function with `#[get("/path")]` and have it become a valid Axum route handler**
So that **I can define HTTP endpoints with minimal boilerplate, like I would in Spring Boot or Rocket**

---

## Description

### Background
This is the hardest and most important story in the entire project. The `#[get]` macro is the core developer-facing API of Autumn. It takes a user's async function, preserves it unchanged, and generates a companion `__autumn_route_info_{name}()` function that pairs the HTTP method and path with an Axum `MethodRouter`. In debug builds, it auto-applies `#[axum::debug_handler]` for better error messages.

The "thin wrapper" macro design (generate around, not into) is the key architectural decision from the brainstorming session. The user's function must remain a valid Rust function independent of the macro — compile errors must point at user code, not generated code.

This story covers only `#[get]` with no extractors and `&'static str` return type. Once this works, other HTTP methods (S-003) are trivial copies. Extractors and complex return types come in later stories.

### Scope
**In scope:**
- `#[get("/path")]` proc macro attribute
- Parsing path string from attribute argument
- Preserving user's async function unchanged
- Generating `__autumn_route_info_{name}()` companion function
- `Route` type definition in `autumn` crate (`method`, `path`, `handler`, `name`)
- Auto-applying `#[cfg_attr(debug_assertions, axum::debug_handler)]`
- Basic `compile_error!` for non-async functions
- `cargo expand` showing understandable generated code

**Out of scope:**
- `#[post]`, `#[put]`, `#[delete]` (S-003)
- Path parameter parsing/extraction (S-006)
- `routes![]` collection macro (S-005)
- Comprehensive `compile_error!` diagnostics (S-007)
- Complex return types (AutumnResult, Markup, Json) — these work because the macro doesn't touch the function body
- `AppState` type (comes with app builder, S-009)

### Sprint Notes
This is the riskiest single story in the project. Sprint 1 is deliberately light (12pts total) to absorb overrun. If proc macro development proves harder than expected, S-025 (AutumnConfig) is the "productive procrastination" task.

- [ ] `#[get("/hello")]` on `async fn hello() -> &'static str { "hello" }` compiles successfully
- [ ] The macro preserves the user's function signature and body unchanged
- [ ] A companion function `__autumn_route_info_hello()` is generated returning `autumn_web::route::Route`

- [ ] `#[get("/hello")]` on a non-function item (struct, enum, etc.) emits a clear error
- [ ] `cargo expand` on a handler shows understandable generated code (user function + route info function)
- [ ] The macro crate does not depend on `axum` or any runtime crate — it generates code referencing `autumn_web::` paths
- [ ] A simple test confirms the macro compiles and the route info function is callable

## User Flow

```rust
// What the developer writes:
use autumn_web::prelude::*;

#[get("/hello")]
async fn hello() -> &'static str {
    "Hello, Autumn!"
}

// What cargo expand shows (approximately):
#[cfg_attr(debug_assertions, axum::debug_handler(state = autumn_web::AppState))]
async fn hello() -> &'static str {
    "Hello, Autumn!"
}

#[doc(hidden)]
pub fn __autumn_route_info_hello() -> autumn_web::route::Route {
    autumn_web::route::Route {
        method: autumn_web::reexports::http::Method::GET,
        path: "/hello",
        handler: autumn_web::reexports::axum::routing::get(hello),
        name: "hello",
    }
}
```

## Technical Notes

### Proc Macro Implementation Plan

The macro lives in `autumn-macros/src/route.rs` with the entry point in `autumn-macros/src/lib.rs`.

**Step 1: Parse attribute arguments**
```rust
// Input: #[get("/hello")]
// Parse the path string from the attribute TokenStream
```

Using `syn` to parse the attribute argument as a `LitStr`.

**Step 2: Parse the annotated item**
```rust
// Verify it's an `async fn` using syn::ItemFn
// Extract: function name, visibility, signature, body
```

**Step 3: Validate**
```rust
// Check: fn.sig.asyncness.is_some() — must be async
// Check: path is not empty
// If validation fails: return compile_error!()
```

**Step 4: Generate output**
```rust
// 1. The original function with debug_handler added
// 2. The __autumn_route_info_{name}() function
```

Using `quote!` to generate the output TokenStream.

### Route Type (in autumn crate)

```rust
// autumn/src/route.rs
use axum::routing::MethodRouter;
use http::Method;

pub struct Route {
    pub method: Method,
    pub path: &'static str,
    pub handler: MethodRouter<AppState>,
    pub name: &'static str,
}
```

Note: The `AppState` type needs to exist for `MethodRouter<AppState>`. For this story, a placeholder `pub struct AppState;` in the autumn crate is sufficient. The real AppState (holding config, db pool) comes in S-009.

### Re-exports Needed (in autumn crate)

The macro generates code referencing `autumn_web::reexports::*` and `autumn_web::route::*`. These paths must exist:

```rust
// autumn/src/lib.rs
pub mod reexports {
    pub use axum;
    pub use http;
}
pub mod route;
```

### Testing Strategy

1. **Compile-pass tests:** A test file that uses `#[get("/test")]` and compiles successfully
2. **Compile-fail tests:** Using `trybuild` crate to test that non-async functions produce the expected `compile_error!`
3. **Route info verification:** Call `__autumn_route_info_{name}()` and assert the method and path are correct
4. **Expansion test:** Use `cargo expand` manually (or `macrotest` crate) to verify the expansion is what we expect

### Dependencies Added

**`autumn-macros/Cargo.toml`:**
- `syn = { version = "2", features = ["full"] }` (already from S-001)
- `quote = "1"` (already from S-001)
- `proc-macro2 = "1"` (already from S-001)

**`autumn/Cargo.toml`:**
- `axum = "0.8"` (already from S-001)
- `http = "1"` (via axum, but explicit for `Method` type)
- `trybuild = "1"` (dev-dependency for compile-fail tests)

### Known Risks

1. **Token span preservation:** Ensuring compile errors point at user code, not generated code. `syn`'s `Span` handling is critical — must use the original function's spans, not generated spans.
2. **`debug_handler` state parameter:** `axum::debug_handler(state = autumn_web::AppState)` requires `AppState` to exist and implement certain traits. May need a minimal stub.
3. **`MethodRouter` generic:** `MethodRouter<AppState>` ties the route to a specific state type. This is intentional (framework controls state) but must be set up correctly.

## Additional Notes

- **Sprint strategy:** Start this immediately after S-001 is green. If proc macro work gets frustrating, switch to S-025 (config struct) for a productive break — it's pure data structures with no macro dependency.
- **Cargo expand is your friend.** Run it early and often to see what the macro actually generates. The output should look like something you'd write by hand.
- **Don't optimize the macro code.** The proc macro runs at compile time, not runtime. Readability and correctness matter more than performance. Use `clone()` freely.
- **The "no extractors" constraint is key.** This story deliberately avoids handlers with parameters like `Path<i32>` or `Db`. Those work because the macro doesn't touch the function signature — but testing them here would pull in dependencies that don't exist yet.
