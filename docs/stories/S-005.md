## User Story

As a **framework user**
I want to **collect annotated route handlers into a `Vec<Route>` with `routes![handler_a, handler_b]`**
So that **I can pass all my routes to the app builder in a single expression, without manually calling hidden functions**

---

## Description

### Background
Each `#[get]`, `#[post]`, etc. macro generates a hidden companion function `__autumn_route_info_{name}()`. Users should never need to know about or call these functions directly. The `routes![]` macro provides a clean, declarative way to collect handlers by name — it transforms each identifier into a call to its companion function and wraps the results in a `Vec<Route>`.

This macro is a `proc_macro` (not `proc_macro_attribute`) since it operates on a token list, not on an annotated item. It lives in its own file (`autumn-macros/src/routes_macro.rs`) for separation of concerns.

### Scope
**In scope:**
- `routes![handler_a, handler_b]` proc_macro
- Parsing comma-separated list of paths (trailing comma supported)
- Transforming last segment: `name` -> `__autumn_route_info_name`
- Module-qualified paths: `routes![users::list]` -> `users::__autumn_route_info_list()`
- Empty `routes![]` returns empty `Vec`
- Compile-fail test for nonexistent handlers
- Re-export from `autumn` crate

**Out of scope:**
- Route conflict detection (comes with app builder, S-009)
- Automatic route discovery (no file-system scanning)
- Route grouping / prefixing (later story)

## Technical Notes

### Implementation

The macro is implemented in `autumn-macros/src/routes_macro.rs` and registered in `autumn-macros/src/lib.rs`.

**Entry point (`lib.rs`):**

```rust
#[proc_macro]
pub fn routes(input: TokenStream) -> TokenStream {
    routes_macro::routes_macro(input.into()).into()
}
```

**Core implementation (`routes_macro.rs`):**

```rust
pub fn routes_macro(input: TokenStream) -> TokenStream {
    // Handle empty input
    if input.is_empty() {
        return quote! { ::std::vec::Vec::new() };
    }

    // Parse comma-separated list of paths (trailing comma ok)
    let paths: Punctuated<Path, Token![,]> =
        match syn::parse::Parser::parse2(Punctuated::parse_terminated, input) {
            Ok(paths) => paths,
            Err(err) => return err.to_compile_error(),
        };

    // Transform each path's last segment to __autumn_route_info_{name}
    let route_info_calls: Vec<_> = paths
        .iter()
        .map(|path| {
            let mut info_path = path.clone();
            if let Some(last) = info_path.segments.last_mut() {
                let info_name = format_ident!("__autumn_route_info_{}", last.ident);
                last.ident = info_name;
            }
            quote! { #info_path() }
        })
        .collect();

    quote! {
        vec![#(#route_info_calls),*]
    }
}
```

**Key design decisions:**
- Uses `Punctuated::parse_terminated` to handle trailing commas naturally
- Clones each path and mutates only the last segment, preserving module prefixes
- Empty input short-circuits to `Vec::new()` without parsing
- Nonexistent handlers fail at link time with a clear "function not found" error

## Additional Notes

- The `routes![]` macro does not validate that the listed functions actually have route attributes — it just generates calls to `__autumn_route_info_{name}()`. If the function exists but wasn't annotated, the user gets a "function not found" error pointing at the `routes![]` call site.
- Module-qualified paths are important for larger applications that organize handlers into modules (e.g., `routes![users::list, users::create, products::index]`).
