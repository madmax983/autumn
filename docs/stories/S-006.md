## User Story

As a **framework user**
I want to **use `Path<T>` in my handler signatures to extract typed URL parameters**
So that **I can build dynamic routes like `/users/{id}` without manual parsing**

---

## Description

### Background
The route macros already pass the user's function signature through unchanged — `Path<i32>` already works because Axum's extractor system handles it. This story is about ensuring the integration is solid: the path pattern in `#[get("/users/{id}")]` lines up with the `Path<i32>` extractor, type mismatches produce clear errors, and the whole flow is tested end-to-end.

Since the macro doesn't modify the function signature, `Path` extraction "just works" via Axum. The value of this story is **verification, documentation, and re-export** — not new macro logic.

### Scope
**In scope:**
- Re-export `axum::extract::Path` from `autumn` (or `autumn_web::prelude`)
- Integration tests: `#[get("/users/{id}")]` with `Path<i32>` compiles
- Integration tests: `#[get("/posts/{year}/{slug}")]` with `Path<(i32, String)>` compiles
- Verify route info function has the correct path pattern
- Test that `Path` extractors work with `debug_handler` in debug builds

**Out of scope:**
- Custom path parsing beyond Axum's built-in behavior
- Path validation at macro expansion time (the macro doesn't inspect function params)
- Form, Query, Json extractors (separate stories)

## Technical Notes

### Re-exports

Add to `autumn/src/lib.rs` (or a future `prelude` module):
```rust
pub mod extract {
    pub use axum::extract::{Path, Query};
}
```

Or simply re-export directly. The key is that users can write `use autumn_web::extract::Path;` without depending on axum directly.

### Testing

```rust
use autumn_web::{get, extract::Path};

#[get("/users/{id}")]
async fn get_user(id: Path<i32>) -> &'static str {
    "user"
}

#[test]
fn path_param_route_has_correct_path() {
    let route = __autumn_route_info_get_user();
    assert_eq!(route.path, "/users/{id}");
}
```

### No Macro Changes Needed

The existing `route_macro()` already preserves the path string as-is and doesn't inspect function parameters. This story is purely additive (re-exports + tests).
