## User Story

As a **framework user**
I want to **have Autumn create and manage a database connection pool automatically**
So that **my handlers can query Postgres without manually setting up connection infrastructure**

## Description

### Background

This is the first story in the database layer epic. It introduces `diesel-async` with `deadpool` as the connection pool manager, wired into the `AppBuilder` lifecycle. After this story, the app creates a pool at startup using `config.database.url` and makes it available via `AppState`.

The `Db` extractor (S-017) will use this pool, but this story focuses on pool creation, health, and lifecycle — not request-level extraction.

### Scope

**In scope:**
- Add `diesel-async` and `deadpool-diesel` dependencies
- `create_pool()` function in `autumn/src/db.rs` that builds a `Pool<AsyncPgConnection>` from `DatabaseConfig`
- Update `AppState` to hold `Pool<AsyncPgConnection>` (optional — `None` when no database URL is configured)
- Wire pool creation into `AppBuilder::run()` — create pool after config load, before router build
- Pool respects `DatabaseConfig.pool_size` and `DatabaseConfig.connect_timeout_secs`
- Skip pool creation if database URL is the default placeholder (no-op when user doesn't configure a DB)
- Startup log: "Database pool created (N connections)" or "No database configured, skipping pool"
- New module: `autumn/src/db.rs`

**Out of scope:**
- `Db` extractor (S-017)
- `#[derive(Model)]` (S-018 — parallel story)
- Diesel migrations or schema management
- Connection retry logic (v1.0)
- Multiple database support (v1.0)

### Pool Lifecycle

```
AppBuilder::run()
  1. Load config
  2. Validate routes
  3. Create DB pool (if database.url is configured)   ← NEW
  4. Build AppState with pool
  5. Build router with state
  6. Serve
```

## Technical Notes

### `autumn/src/db.rs`

```rust
//! Database connection pool management.

use diesel_async::AsyncPgConnection;
use diesel_async::pooled_connection::deadpool::Pool;
use diesel_async::pooled_connection::AsyncDieselConnectionManager;

use crate::config::DatabaseConfig;

/// Create a connection pool from the database configuration.
///
/// Returns `None` if no database URL is configured (default placeholder).
///
/// # Errors
///
/// Returns an error if the pool cannot be created (e.g., invalid URL format).
pub fn create_pool(config: &DatabaseConfig) -> Result<Option<Pool<AsyncPgConnection>>, PoolError> {
    // Skip pool creation for the default placeholder URL
    if config.url == "postgres://localhost/autumn_dev" && !has_explicit_db_config() {
        return Ok(None);
    }

    let manager = AsyncDieselConnectionManager::<AsyncPgConnection>::new(&config.url);
    let pool = Pool::builder(manager)
        .max_size(config.pool_size)
        .build()
        .map_err(PoolError::Build)?;

    Ok(Some(pool))
}
```

**Design decision:** Pool creation is synchronous (deadpool builds the pool eagerly but doesn't connect until first use). Connection validation happens on first `pool.get()` call. This means `create_pool()` succeeds even if the database is unreachable — failures surface when handlers try to use the pool.

### Detecting "no explicit config"

The default `DatabaseConfig::url` is `"postgres://localhost/autumn_dev"`. We need a way to distinguish "user didn't configure a database" from "user explicitly set this URL". Options:

1. **Check if `autumn.toml` has a `[database]` section** — complex, requires re-parsing
2. **Make the URL an `Option<String>`** — cleanest, but breaks existing config deserialization
3. **Use a sentinel value** — check if URL equals the default AND no env var override

**Recommended:** Option 2 — change `DatabaseConfig.url` to `Option<String>` with `None` as the default. If `None`, skip pool creation. This is the cleanest signal for "user didn't configure a database."

This is a small breaking change to the config struct, but since it's pre-v0.1 and only used internally by `AppBuilder::run()`, it's safe.

### Updated AppState

```rust
#[derive(Clone, Debug)]
pub struct AppState {
    pub pool: Option<Pool<AsyncPgConnection>>,
}
```

The `Db` extractor (S-017) will return a 503 Service Unavailable if it tries to extract from `AppState` when `pool` is `None`.

### Wiring into AppBuilder

In `autumn/src/app.rs`:

```rust
use crate::db;

pub async fn run(self) {
    let config = AutumnConfig::load().unwrap_or_else(|e| { ... });

    // ... validate routes ...

    // Create database pool
    let pool = db::create_pool(&config.database).unwrap_or_else(|e| {
        eprintln!("Failed to create database pool: {e}");
        std::process::exit(1);
    });

    if pool.is_some() {
        println!("  Database pool: {} max connections", config.database.pool_size);
    } else {
        println!("  Database: not configured");
    }

    // Build router with state
    let state = AppState { pool };
    let router = router.layer(RequestIdLayer).with_state(state);
    // ...
}
```

### Testing Strategy

Pool creation tests can run without a live database:

```rust
#[test]
fn create_pool_with_none_url_returns_none() {
    let config = DatabaseConfig { url: None, ..Default::default() };
    let pool = create_pool(&config).unwrap();
    assert!(pool.is_none());
}

#[test]
fn create_pool_with_url_returns_some() {
    let config = DatabaseConfig {
        url: Some("postgres://localhost/test".into()),
        ..Default::default()
    };
    let pool = create_pool(&config).unwrap();
    assert!(pool.is_some());
}
```

Integration tests with a real database are deferred to S-017 (Db extractor) and CI setup.

## Additional Notes

- `deadpool` was chosen over `bb8` per architecture doc open issue #1. Deadpool has a simpler API and better integration with diesel-async's built-in pooled connection module.
- Pool creation is synchronous — it builds the pool config but doesn't establish connections. First connection happens on first `pool.get()`. This means the server starts quickly even with a slow database.
- Connection retry and health monitoring are v1.0 features. For v0.1, if the database is unreachable, handlers get errors when they try to use `Db`.
