## User Story

As a **framework user**
I want to **return `Markup` from handlers and use `html!{}` templates without depending on maud directly**
So that **I can render HTML pages with zero configuration and a single `use autumn::prelude::*`**

---

## Description

### Background

Maud is Autumn's chosen HTML templating engine — compile-time HTML generation via Rust macros. Autumn re-exports `maud::Markup`, `maud::html!`, and `maud::PreEscaped` so users don't need `maud` as a direct dependency. Since `Markup` implements Axum's `IntoResponse`, it already works as a handler return type — this story is about the re-export and verification.

After this story:
```rust
use autumn::prelude::*; // includes html!, Markup

#[get("/")]
async fn index() -> Markup {
    html! {
        h1 { "Welcome to Autumn" }
        p { "It's " (chrono::Utc::now()) " somewhere." }
    }
}
```

## Technical Notes

### Dependencies

Add to workspace `Cargo.toml`:
```toml
maud = { version = "0.26", features = ["axum"] }
```

The `axum` feature enables `IntoResponse` for `Markup`. This is critical — without it, `Markup` can't be a handler return type.

Add to `autumn/Cargo.toml`:
```toml
maud.workspace = true
```

### Re-exports

In `autumn/src/lib.rs` or a new `autumn/src/render.rs`:

```rust
// Re-export Maud types for HTML rendering
pub use maud::{html, Markup, PreEscaped};
```

**Decision:** Put directly in `lib.rs` for now. A `render.rs` module makes sense when we have multiple rendering backends, but for v0.1 Maud is the only one.

### Verify `IntoResponse`

Maud's `Markup` implements `IntoResponse` via the `axum` feature flag. This means:
- Response content type is `text/html; charset=utf-8`
- Response body is the rendered HTML string

Test this:
```rust
#[tokio::test]
async fn maud_handler_returns_html() {
    use autumn::{get, html, Markup};

    #[get("/")]
    async fn index() -> Markup {
        html! { h1 { "hello" } }
    }

    let app = Router::new()
        .route("/", axum::routing::get(index))
        // No state needed for this simple test

    let response = app
        .oneshot(Request::builder().uri("/").body(Body::empty()).unwrap())
        .await
        .unwrap();

    assert_eq!(response.status(), StatusCode::OK);

    let content_type = response.headers()
        .get("content-type")
        .unwrap()
        .to_str()
        .unwrap();
    assert!(content_type.contains("text/html"));

    let body = axum::body::to_bytes(response.into_body(), usize::MAX)
        .await
        .unwrap();
    let html = String::from_utf8(body.to_vec()).unwrap();
    assert!(html.contains("<h1>hello</h1>"));
}
```

### Compile-Pass Test

Add `autumn/tests/compile-pass/maud_handler.rs`:
```rust
use autumn::{get, html, Markup, AutumnResult};

#[get("/")]
async fn index() -> Markup {
    html! { h1 { "hello" } }
}

#[get("/fallible")]
async fn fallible() -> AutumnResult<Markup> {
    Ok(html! { p { "ok" } })
}

fn main() {}
```

Maud 0.26 is the latest as of early 2026. It supports Axum 0.8 via the `axum` feature flag. Verify compatibility before pinning the version.

## Additional Notes

- Maud generates HTML at compile time — `html! { ... }` produces a `Markup` value that is a pre-rendered string. This is zero-cost at runtime.
- The `PreEscaped` type allows injecting raw HTML strings. Re-export it for advanced use cases (embedding pre-built HTML fragments, inline SVGs, etc.).
- Maud's `html!` macro is a proc macro that creates a `Markup` — we re-export the macro itself, not a wrapper. Users write standard Maud syntax.
- Future: `autumn::render` module could hold layout helpers, component patterns, etc. For v0.1, just the re-exports.
