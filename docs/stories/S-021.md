## User Story

As a **framework user**
I want **`cargo build` to automatically compile Tailwind CSS from my Maud templates**
So that **CSS classes used in `html!{}` produce a tree-shaken stylesheet without manual build steps**

---

## Description

### Background

Autumn's rendering stack is Maud + Tailwind + htmx. Tailwind scans source files for CSS class names and outputs only the CSS actually used. Since Maud templates are Rust code (`html! { div class="bg-blue-500 p-4" { ... } }`), Tailwind must scan `src/**/*.rs` for class strings.

The architecture specifies that `build.rs` is a **template file** — it lives in the user's project (generated by `autumn new` in Sprint 9), not in the `autumn` crate itself. For v0.1, this story creates the `build.rs` template as a file in the Autumn repo that can be copied into projects, and provides a helper function for locating the Tailwind CLI.

The Tailwind CLI is a standalone binary — no Node.js required. It's found at `target/autumn/tailwindcss` (downloaded by `autumn setup`) or on PATH.

After this story, a user project with `build.rs` and Tailwind CLI installed will:
1. Scan `src/**/*.rs` for Tailwind class names on every `cargo build`
2. Output tree-shaken, minified CSS to `static/css/autumn.css`
3. Emit `compile_error!` if the Tailwind CLI is not found

### Scope

**In scope:**
- `build.rs` template file at `autumn/templates/build.rs` (or similar location)
- `find_tailwind_cli()` function: checks `target/autumn/tailwindcss`, then PATH, then fails with clear message
- Tailwind CLI invocation: `-i input.css -o static/css/autumn.css --content src/**/*.rs --minify`
- `cargo:rerun-if-changed` directives for correct incremental rebuilds
- Unit/integration test verifying the template compiles
- Documentation of the build.rs template and its requirements

**Out of scope:**
- `autumn new` project generation (S-035, Sprint 9)
- `autumn setup` CLI for downloading Tailwind (S-036, Sprint 10)
- Tailwind configuration file (`tailwind.config.js`) — use CLI defaults for v0.1
- Feature flag to disable Tailwind (S-044, Sprint 11)
- Tailwind v4 custom content scanner plugins

## Technical Notes

### Template Location

Place at `autumn/templates/build.rs.template`:

```rust
//! Autumn build script — compiles Tailwind CSS from Maud templates.
//!
//! This file was generated by `autumn new`. You can customize it,
//! but the core pipeline (find CLI → scan sources → output CSS) should
//! remain intact for Tailwind integration to work.

use std::path::PathBuf;
use std::process::Command;

fn main() {
    println!("cargo:rerun-if-changed=src/");
    println!("cargo:rerun-if-changed=static/css/input.css");

    // Ensure output directory exists
    std::fs::create_dir_all("static/css").expect("Failed to create static/css directory");

    let tailwind = find_tailwind_cli();

    let output = Command::new(&tailwind)
        .args([
            "-i", "static/css/input.css",
            "-o", "static/css/autumn.css",
            "--content", "src/**/*.rs",
            "--minify",
        ])
        .output()
        .expect("Failed to execute Tailwind CLI");

    if !output.status.success() {
        let stderr = String::from_utf8_lossy(&output.stderr);
        panic!(
            "\n\nTailwind CSS compilation failed:\n{}\n\n\
             Check your static/css/input.css and source files for issues.\n",
            stderr
        );
    }
}

fn find_tailwind_cli() -> PathBuf {
    // 1. Check target/autumn/tailwindcss (downloaded by `autumn setup`)
    let local = PathBuf::from("target/autumn/tailwindcss");
    if local.exists() {
        return local;
    }

    // Windows variant
    let local_exe = PathBuf::from("target/autumn/tailwindcss.exe");
    if local_exe.exists() {
        return local_exe;
    }

    // 2. Check PATH
    if let Ok(path) = which::which("tailwindcss") {
        return path;
    }

    // 3. Fail with clear message
    panic!(
        "\n\nTailwind CSS CLI not found!\n\n\
         Install it using one of:\n\
         1. Run `autumn setup` to download it automatically\n\
         2. Download from: https://github.com/tailwindlabs/tailwindcss/releases\n\
         3. Add `tailwindcss` to your PATH\n\n\
         To disable Tailwind, use:\n\
         autumn = {{ default-features = false }}\n"
    );
}
```

### Why a Template, Not a Crate Feature

The architecture document (Build Pipeline section) explicitly states: "The `build.rs` is a template file generated by `autumn new` into each project. It is NOT part of the `autumn` crate itself."

Reasons:
- `build.rs` runs in the **user's project** context, not Autumn's
- Users may need to customize content scanning paths
- Build scripts can't be "inherited" from dependencies in Cargo
- Keeps the `autumn` crate focused on runtime, not build tooling

### `which` Crate

The template uses `which` for PATH lookup. This must be a build-dependency in the user's project:

```toml
[build-dependencies]
which = "7"
```

This will be automatically added by `autumn new` (S-035). For now, document it.

### Tailwind v4 Compatibility

Tailwind v4 standalone CLI supports `--content` glob patterns. Maud templates use string literals for class names (`class="bg-blue-500 p-4"`), which Tailwind's default content scanner can detect. No custom extractor is needed for basic usage.

**Known limitation:** Dynamic class construction (`format!("bg-{}-500", color)`) won't be detected by Tailwind. This is a fundamental Tailwind limitation, not an Autumn issue. Document it.

### Incremental Builds

`cargo:rerun-if-changed=src/` tells Cargo to re-run build.rs when any file in `src/` changes. This is coarse-grained but correct — Tailwind needs to rescan all source files for class changes.
