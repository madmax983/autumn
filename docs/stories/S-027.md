## User Story

As a **framework user deploying to production**
I want **every config field to be overridable via `AUTUMN_*` environment variables**
So that **I can configure my app per-environment (staging, production, CI) without editing `autumn.toml`**

---

## Description

### Background

Autumn's three-layer configuration system is designed as:

1. Framework defaults (compiled in)
2. `autumn.toml` file overrides
3. `AUTUMN_*` environment variable overrides (highest priority)

Layers 1 and 2 are complete (S-025, S-026). Layer 3 currently only covers three database fields (`AUTUMN_DATABASE__URL`, `AUTUMN_DATABASE__POOL_SIZE`, `AUTUMN_DATABASE__CONNECT_TIMEOUT_SECS`) — implemented as a bootstrap in S-019.

This story expands env var overrides to cover **every** config field, completing the configuration system. The `AUTUMN_` prefix with `__` nesting separator is already established:

```
AUTUMN_SERVER__PORT=8080
AUTUMN_SERVER__HOST=0.0.0.0
AUTUMN_SERVER__SHUTDOWN_TIMEOUT_SECS=60
AUTUMN_LOG__LEVEL=debug
AUTUMN_LOG__FORMAT=Json
AUTUMN_HEALTH__PATH=/healthz
```

After this story, a zero-config Autumn app can be fully configured via environment variables alone — the standard practice for 12-factor apps and container deployments.

### Scope

**In scope:**
- Expand `apply_env_overrides()` in `config.rs` to cover all `ServerConfig`, `LogConfig`, and `HealthConfig` fields
- Follow the existing `AUTUMN_SECTION__FIELD` naming pattern
- Parse numeric fields with validation (warn on invalid values, keep default)
- Parse `LogFormat` enum from string ("Auto", "Pretty", "Json")
- Warn on invalid values via `eprintln!` (consistent with existing behavior)
- Unit tests for every new override: happy path + invalid input
- Document all supported env vars in doc comments

**Out of scope:**
- Generic/reflective env var parsing (manual per-field is fine for 10 fields)
- Nested config beyond two levels (no `AUTUMN_DATABASE__POOL__MAX_SIZE`)
- `.env` file loading (user-land, e.g. `dotenvy`)
- Custom prefix configuration (always `AUTUMN_`)
- Validation across fields (e.g. "if X is set, Y must also be set")

## Technical Notes

### Expanding `apply_env_overrides()`

The existing pattern in `config.rs:111-131` handles database overrides. Extend with the same pattern:

```rust
pub fn apply_env_overrides(&mut self) {
    // ── Server ──
    if let Ok(val) = std::env::var("AUTUMN_SERVER__PORT") {
        match val.parse::<u16>() {
            Ok(port) => self.server.port = port,
            Err(_) => eprintln!(
                "Warning: AUTUMN_SERVER__PORT={val:?} is not a valid port, ignoring"
            ),
        }
    }
    if let Ok(val) = std::env::var("AUTUMN_SERVER__HOST") {
        self.server.host = val;
    }
    if let Ok(val) = std::env::var("AUTUMN_SERVER__SHUTDOWN_TIMEOUT_SECS") {
        match val.parse::<u64>() {
            Ok(secs) => self.server.shutdown_timeout_secs = secs,
            Err(_) => eprintln!(
                "Warning: AUTUMN_SERVER__SHUTDOWN_TIMEOUT_SECS={val:?} is not valid, ignoring"
            ),
        }
    }

    // ── Log ──
    if let Ok(val) = std::env::var("AUTUMN_LOG__LEVEL") {
        self.log.level = val;
    }
    if let Ok(val) = std::env::var("AUTUMN_LOG__FORMAT") {
        match val.as_str() {
            "Auto" => self.log.format = LogFormat::Auto,
            "Pretty" => self.log.format = LogFormat::Pretty,
            "Json" => self.log.format = LogFormat::Json,
            _ => eprintln!(
                "Warning: AUTUMN_LOG__FORMAT={val:?} is not valid \
                 (expected Auto, Pretty, or Json), ignoring"
            ),
        }
    }

    // ── Health ──
    if let Ok(val) = std::env::var("AUTUMN_HEALTH__PATH") {
        self.health.path = val;
    }

    // ── Database (existing, unchanged) ──
    // ... existing code ...
}
```

### LogFormat Parsing

`LogFormat` needs a `from_str` or manual match. Since there are only 3 variants and `Deserialize` is already implemented, a simple match is clearest. Don't over-abstract this.

### Testing Pattern

Use the existing `EnvGuard` RAII helper in `config.rs` tests (line 299-324). Each new test follows the same shape:

```rust
#[test]
fn env_override_server_port() {
    let _guard = EnvGuard::set("AUTUMN_SERVER__PORT", "8080");
    let mut config = AutumnConfig::default();
    config.apply_env_overrides();
    assert_eq!(config.server.port, 8080);
}
```

### Precedence Verification

Add one test that verifies env vars override TOML values:

```rust
#[test]
fn env_overrides_toml_values() {
    let _guard = EnvGuard::set("AUTUMN_SERVER__PORT", "9999");
    let dir = tempfile::tempdir().unwrap();
    let path = dir.path().join("autumn.toml");
    std::fs::write(&path, "[server]\nport = 4000\n").unwrap();
    let mut config = AutumnConfig::load_from(&path).unwrap();
    config.apply_env_overrides();
    assert_eq!(config.server.port, 9999); // env wins
}
```

## Additional Notes

- The `EnvGuard` pattern uses `unsafe` env var mutation. Tests should use unique env var names or run with `--test-threads=1` if parallelism causes flakes. Currently tests use the real `AUTUMN_*` names, which is fine for single-threaded test runs.
- Future enhancement: consider using `tracing::warn!` instead of `eprintln!` for override warnings once S-028 (structured logging) is in place. For now, `eprintln!` is correct since logging may not be initialized yet when config loads.
- The `AUTUMN_ENV` variable (used by `LogFormat::Auto` in S-028) is intentionally *not* part of the config struct — it's a meta-variable that affects behavior, not a config field.
