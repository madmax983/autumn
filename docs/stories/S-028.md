## User Story

As a **framework user running an Autumn app in production**
I want **structured, leveled logging with request IDs in every log line**
So that **I can debug issues, trace requests, and feed logs to observability tools (Datadog, Grafana Loki, etc.)**

---

## Description

### Background

Autumn currently uses `println!` / `eprintln!` for all output — startup banner, route mounting, errors. This is fine for development but inadequate for production:

- No log levels (can't filter noise)
- No structured output (can't parse with log aggregators)
- No request context (can't correlate logs to individual requests)

This story replaces `println!` with `tracing` and sets up `tracing-subscriber` based on config:

```rust
// After this story, all framework output uses tracing:
tracing::info!("Autumn v{}", env!("CARGO_PKG_VERSION"));
tracing::info!(port = config.server.port, "Listening");
tracing::debug!(method = %route.method, path = route.path, "Mounted route");

// And in handlers, request_id is automatically in scope:
#[get("/api/users")]
async fn list_users() -> Json<Vec<User>> {
    tracing::info!("Fetching users");
    // Log output: 2026-07-01T12:00:00Z INFO request_id=abc123 list_users: Fetching users
    // ...
}
```

The `LogConfig` struct and `LogFormat` enum already exist in `config.rs` — this story wires them up.

### Scope

## Technical Notes

### New Module: `autumn/src/logging.rs`

```rust
use crate::config::{LogConfig, LogFormat};
use tracing_subscriber::{EnvFilter, fmt, layer::SubscriberExt, util::SubscriberInitExt};

/// Initialize the tracing subscriber based on configuration.
///
/// Must be called once, early in application startup. Panics if
/// called twice (tracing global subscriber is set-once).
pub fn init(config: &LogConfig) {
    let filter = EnvFilter::try_new(&config.level)
        .unwrap_or_else(|e| {
            eprintln!("Warning: invalid log level {:?}: {e}, falling back to \"info\"", config.level);
            EnvFilter::new("info")
        });

    let format = match config.format {
        LogFormat::Auto => is_production(),
        LogFormat::Pretty => false,
        LogFormat::Json => true,
    };

    if format {
        // JSON output for production / log aggregators
        tracing_subscriber::registry()
            .with(filter)
            .with(fmt::layer().json())
            .init();
    } else {
        // Pretty output for development
        tracing_subscriber::registry()
            .with(filter)
            .with(fmt::layer().pretty())
            .init();
    }
}

fn is_production() -> bool {
    std::env::var("AUTUMN_ENV")
        .map(|v| v.eq_ignore_ascii_case("production"))
        .unwrap_or(false)
}
```

### Request Span Integration

The existing `RequestIdLayer` (in `middleware.rs`, S-011) generates a UUID and inserts it into request extensions. To wire this into tracing spans, add a `TraceLayer` from `tower-http`:

```rust
use tower_http::trace::TraceLayer;

// In app.rs, after RequestIdLayer:
let router = router
    .layer(
        TraceLayer::new_for_http()
            .make_span_with(|req: &http::Request<_>| {
                let request_id = req.extensions()
                    .get::<crate::middleware::RequestId>()
                    .map(|id| id.to_string())
                    .unwrap_or_default();
                tracing::info_span!(
                    "request",
                    method = %req.method(),
                    path = %req.uri().path(),
                    request_id = %request_id,
                )
            })
    )
    .layer(RequestIdLayer);
```

**Layer ordering matters:** `RequestIdLayer` must run first (outermost) so the request ID exists when `TraceLayer` creates the span. In Axum, layers are applied bottom-to-top, so `RequestIdLayer` should be added *after* `TraceLayer` in the builder chain.

### Replacing println! in app.rs

Current `println!` calls and their replacements:

| Current | Replacement |
|---------|-------------|
| `println!("Autumn v{}", ...)` | `tracing::info!("Autumn v{}", ...)` |
| `println!("  Database pool: ...")` | `tracing::info!(max_connections = ..., "Database pool configured")` |
| `println!("  Database: not configured")` | `tracing::info!("Database: not configured")` |
| `println!("  {} {} ({})", method, path, name)` | `tracing::debug!(method = %method, path, name, "Mounted route")` |
| `println!("Listening on http://...")` | `tracing::info!(addr = %addr, "Listening")` |
| `eprintln!("Failed to load ...")` | `tracing::error!(...)` or keep `eprintln!` (pre-logging-init) |

**Note:** Config loading errors happen *before* logging is initialized. These must stay as `eprintln!` — tracing isn't available yet.

### Dependencies to Add

In `autumn/Cargo.toml`:
```toml

```

`tower-http` already exists as a dependency; ensure the `trace` feature is enabled:
```toml

```

### Initialization Order in `app.rs`

```rust
pub async fn run(self) {
    // 1. Load configuration (errors use eprintln!, can't use tracing yet)
    let config = AutumnConfig::load().unwrap_or_else(|e| { ... });

    // 2. Initialize logging IMMEDIATELY after config loads
    crate::logging::init(&config.log);

    // 3. Everything after this uses tracing
    tracing::info!("Autumn v{}", env!("CARGO_PKG_VERSION"));
    // ...
}
```

## Additional Notes

- `tracing::info!` output in tests can be noisy. Consider using `tracing_test` crate or `tracing_subscriber::fmt::TestWriter` in test helpers. Not required for this story.
- The `EnvFilter` supports `RUST_LOG`-style directives. If both `config.log.level` and `RUST_LOG` are set, the config value wins (since we pass it directly to `EnvFilter::try_new`). `RUST_LOG` could be supported as a fallback in a future enhancement.
- JSON output from `tracing-subscriber` includes timestamp, level, target, span context, and message by default — no custom formatting needed for v0.1.
