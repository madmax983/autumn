## User Story

As a **platform engineer deploying an Autumn app behind a load balancer**
I want **a `/health` endpoint that reports application and database pool status**
So that **my load balancer, Kubernetes liveness probe, or monitoring system can determine if the app is healthy**

---

## Description

### Background

Every production web application needs a health check endpoint. Load balancers use it to decide which instances receive traffic. Kubernetes uses it for liveness/readiness probes. Monitoring systems use it to alert on degraded services.

Autumn provides this out of the box — no user configuration required. The health endpoint:

- Returns JSON with app status, version, and database pool metrics
- Returns `200 OK` when healthy, `503 Service Unavailable` when degraded
- Is automatically mounted at the configured path (default: `/health`)
- Works even when no database is configured (just omits pool info)

```json
// GET /health → 200 OK
{
  "status": "ok",
  "version": "0.1.0",
  "pool": {
    "size": 10,
    "available": 8,
    "waiting": 0
  }
}

// GET /health → 503 Service Unavailable (pool exhausted)
{
  "status": "degraded",
  "version": "0.1.0",
  "pool": {
    "size": 10,
    "available": 0,
    "waiting": 5
  }
}

// GET /health → 200 OK (no database configured)
{
  "status": "ok",
  "version": "0.1.0"
}
```

### Scope

**In scope:**
- Create `autumn/src/health.rs` module with the health check handler
- JSON response with `status`, `version`, and optional `pool` object
- `200 OK` when healthy, `503 Service Unavailable` when degraded
- Pool is degraded when `available == 0` and `waiting > 0`
- Mount automatically in the app builder at `config.health.path`
- Handle the no-database case gracefully (omit pool, always healthy)
- Integration test with a mock/test router

**Out of scope:**
- Custom health check callbacks (user-defined checks)
- Readiness vs liveness distinction (single endpoint for v0.1)
- HTML health page (JSON only)
- Authentication on the health endpoint
- Response caching or rate limiting
- Detailed diagnostics (uptime, memory usage, etc.)

## Technical Notes

### New Module: `autumn/src/health.rs`

```rust
use axum::extract::State;
use axum::http::StatusCode;
use axum::response::IntoResponse;
use axum::Json;

use crate::AppState;

/// Health check handler.
///
/// Returns pool status when a database is configured, or a simple
/// "ok" response when running without a database.
pub async fn handler(State(state): State<AppState>) -> impl IntoResponse {
    match &state.pool {
        Some(pool) => {
            let status = pool.status();
            let available = status.available as u64;
            let size = status.max_size as u64;
            let waiting = status.waiting as u64;

            let healthy = available > 0 || waiting == 0;

            let body = serde_json::json!({
                "status": if healthy { "ok" } else { "degraded" },
                "version": env!("CARGO_PKG_VERSION"),
                "pool": {
                    "size": size,
                    "available": available,
                    "waiting": waiting,
                }
            });

            let status_code = if healthy {
                StatusCode::OK
            } else {
                StatusCode::SERVICE_UNAVAILABLE
            };

            (status_code, Json(body))
        }
        None => {
            let body = serde_json::json!({
                "status": "ok",
                "version": env!("CARGO_PKG_VERSION"),
            });
            (StatusCode::OK, Json(body))
        }
    }
}
```

### Pool Status from Deadpool

`deadpool::managed::Pool::status()` returns a `Status` struct:

```rust
pub struct Status {
    pub max_size: usize,
    pub size: usize,       // current pool size (may be less than max during warmup)
    pub available: usize,   // idle connections ready to use
    pub waiting: usize,     // tasks waiting for a connection
}
```

The "degraded" heuristic: `available == 0 && waiting > 0` means all connections are in use *and* requests are queuing. `available == 0 && waiting == 0` just means all connections are active but nothing is blocked yet.

### Mounting in App Builder

In `app.rs`, after user routes and before framework routes:

```rust
// Health check endpoint (auto-mounted)
let health_path = config.health.path.clone();
router = router.route(&health_path, axum::routing::get(crate::health::handler));
tracing::debug!(path = %health_path, "Mounted health check");
```

Mount the health check *before* user routes in the router to ensure it can't be accidentally overridden by a user route at the same path. Alternatively, log a warning if a user route conflicts with the health path.

### No-Database Case

`AppState.pool` is already `Option<Pool>`. The handler pattern-matches on `Some`/`None`, so the no-database path is naturally handled. The response omits the `pool` field entirely rather than returning `null` — cleaner for consumers.

### Testing

Test with a router that has `AppState { pool: None }`:

```rust
#[tokio::test]
async fn health_no_database() {
    let app = axum::Router::new()
        .route("/health", axum::routing::get(crate::health::handler))
        .with_state(AppState { pool: None });

    let response = app
        .oneshot(Request::get("/health").body(Body::empty()).unwrap())
        .await
        .unwrap();

    assert_eq!(response.status(), StatusCode::OK);

    let body: serde_json::Value = /* deserialize */;
    assert_eq!(body["status"], "ok");
    assert!(body.get("pool").is_none());
}
```

Testing the degraded path requires a real pool or a mock — consider using a pool with `max_size=1` and holding a connection to exhaust it.

## Additional Notes

- `serde_json` is already a dependency (used in config tests). No new dep needed for JSON serialization in the handler.
- The `version` field uses `env!("CARGO_PKG_VERSION")` which resolves at compile time to the *autumn crate* version. This is the framework version, not the user's app version. A future enhancement could include the app's version via config.
- Kubernetes typically expects `/healthz` (with the 'z'). Autumn defaults to `/health` but this is configurable via `config.health.path` or `AUTUMN_HEALTH__PATH` (after S-027).
