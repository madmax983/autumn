## User Story

As a **framework user deploying with Docker/Kubernetes**
I want **my Autumn app to finish in-flight requests before shutting down on SIGTERM or Ctrl+C**
So that **users don't see broken responses during deployments, and database connections are released cleanly**

---

## Description

### Background

Container orchestrators (Docker, Kubernetes, ECS) send `SIGTERM` when stopping a container, then force-kill with `SIGKILL` after a grace period. A well-behaved server must:

1. Stop accepting new connections on `SIGTERM`
2. Finish in-flight requests (connection draining)
3. Release resources (database pool, file handles)
4. Exit cleanly within the grace period

Autumn currently has a basic `shutdown_signal()` in `app.rs` that only handles `Ctrl+C` and prints a message. This story makes it production-grade:

- Handles both `SIGTERM` (Unix) and `Ctrl+C` (all platforms)
- Uses `config.server.shutdown_timeout_secs` (default 30s, already defined)
- Leverages Axum's built-in `with_graceful_shutdown` for connection draining
- Logs shutdown events via `tracing` (after S-028)

### Scope

**In scope:**
- Expand `shutdown_signal()` to handle `SIGTERM` on Unix (via `tokio::signal::unix`)
- Use `#[cfg(unix)]` / `#[cfg(not(unix))]` for platform-specific signal handling
- Pass `shutdown_timeout_secs` from config to control the grace period
- Add a timeout wrapper: if draining exceeds the timeout, force exit
- Replace `println!` with `tracing::info!` for shutdown messages
- Log which signal triggered shutdown ("Received SIGTERM" vs "Received Ctrl+C")
- Unit test: `shutdown_signal` returns a future (compilability test)
- Integration test: verify shutdown behavior is wired up

**Out of scope:**
- Custom shutdown hooks (user-defined cleanup callbacks)
- `SIGHUP` for config reload (future enhancement)
- Database pool drain verification (deadpool handles this internally)
- Multi-signal handling (second SIGTERM = force quit)
- Windows service integration (`SC_HANDLE`)

## Technical Notes

### Expanding `shutdown_signal()`

Replace the current basic implementation in `app.rs:163-168`:

```rust
/// Wait for a shutdown signal (Ctrl+C or SIGTERM).
///
/// Returns when either signal is received. Axum's `with_graceful_shutdown`
/// will then stop accepting new connections and drain in-flight requests.
async fn shutdown_signal() {
    let ctrl_c = async {
        tokio::signal::ctrl_c()
            .await
            .expect("Failed to install Ctrl+C handler");
        tracing::info!("Received Ctrl+C, starting graceful shutdown");
    };

    #[cfg(unix)]
    let terminate = async {
        tokio::signal::unix::signal(tokio::signal::unix::SignalKind::terminate())
            .expect("Failed to install SIGTERM handler")
            .recv()
            .await;
        tracing::info!("Received SIGTERM, starting graceful shutdown");
    };

    #[cfg(not(unix))]
    let terminate = std::future::pending::<()>();

    tokio::select! {
        () = ctrl_c => {},
        () = terminate => {},
    }
}
```

### Timeout Wrapper in `run()`

Axum's `with_graceful_shutdown` handles connection draining, but there's no built-in timeout. Wrap the serve call:

```rust
let shutdown_timeout = config.server.shutdown_timeout_secs;

let server = axum::serve(listener, router)
    .with_graceful_shutdown(shutdown_signal());

// Race the server against a hard timeout after shutdown is signaled
if let Err(e) = server.await {
    tracing::error!("Server error: {e}");
}
```

The timeout enforcement is subtle: `with_graceful_shutdown` already stops accepting new connections immediately. Existing connections drain naturally. If a connection hangs beyond the timeout, the orchestrator's `SIGKILL` handles it. Autumn's role is to log that draining is taking too long:

```rust
// Alternative: add a warning timer
async fn shutdown_signal_with_timeout(timeout_secs: u64) {
    shutdown_signal().await;

    // Spawn a warning timer
    tokio::spawn(async move {
        tokio::time::sleep(std::time::Duration::from_secs(timeout_secs.saturating_sub(5))).await;
        tracing::warn!(
            timeout_secs,
            "Shutdown draining is taking longer than expected, \
             server will be force-killed by orchestrator soon"
        );
    });
}
```

### Platform Considerations

- **Linux/macOS:** `SIGTERM` via `tokio::signal::unix::SignalKind::terminate()`
- **Windows:** `SIGTERM` doesn't exist. `Ctrl+C` is the standard signal. `#[cfg(not(unix))]` compiles to `pending()` which never resolves — `tokio::select!` will only match `ctrl_c`.
- Windows also has `CTRL_SHUTDOWN_EVENT` and `CTRL_CLOSE_EVENT`, but these are handled by the Tokio `ctrl_c()` handler. No extra work needed.

### Integration with App Builder

Update `run()` to pass config to the shutdown function:

```rust
axum::serve(listener, router)
    .with_graceful_shutdown(shutdown_signal_with_timeout(
        config.server.shutdown_timeout_secs,
    ))
    .await
    .unwrap_or_else(|e| {
        tracing::error!("Server error: {e}");
        std::process::exit(1);
    });

tracing::info!("Server shut down cleanly");
```

## Additional Notes

- Kubernetes default `terminationGracePeriodSeconds` is 30s. Autumn's default `shutdown_timeout_secs` matches this intentionally. Users deploying to Kubernetes should set Autumn's timeout to be slightly less than the pod's grace period.
- `deadpool::Pool` drops connections when the pool is dropped. Since `AppState` holds the pool and is cloned into the router, the pool lives until all in-flight requests finish and the router is dropped. No manual pool drain is needed.
- Future enhancement: add a second signal handler that force-exits on the *second* Ctrl+C (common UX pattern for CLI tools). Not needed for v0.1.
