## User Story

As a **framework user**
I want **a single `use autumn::prelude::*;` import**
So that **I don't need to remember and maintain 5+ individual imports for every handler file**

---

## Description

### Background

Currently, an Autumn handler file requires multiple imports:

```rust
use autumn::{get, post, html, Markup, Json, Db};
use autumn::error::AutumnResult;
// maybe more...
```

This is friction — especially for new users who don't know what's available. A `prelude` module collects the most commonly used types and macros into a single glob import:

```rust
use autumn::prelude::*;

#[get("/")]
async fn index(db: Db) -> AutumnResult<Markup> {
    // Everything just works — get, Db, AutumnResult, Markup all in scope
    let users = db::list_users(&db).await?;
    Ok(html! {
        h1 { "Users: " (users.len()) }
    })
}
```

The prelude should include types that *most* handlers need, while keeping niche types (config internals, middleware, route structs) out.

### Scope

**In scope:**
- Create `autumn/src/prelude.rs` module
- Re-export route macros: `get`, `post`, `put`, `delete`
- Re-export the main macro: `main`
- Re-export collection macro: `routes`
- Re-export rendering types: `Markup`, `PreEscaped`, `html`
- Re-export extractors: `Db`, `Json`, `Form`
- Re-export error types: `AutumnError`, `AutumnResult`
- Re-export `AppState` (needed for custom extractors)
- Declare module in `lib.rs`
- Doc comments explaining what's included and why

**Out of scope:**
- Re-exporting Axum internals (users reach into `autumn::reexports::axum`)
- Re-exporting config types (only needed in specialized code)
- Re-exporting middleware types (framework-internal)
- Replacing existing root-level exports (keep backward compatibility)
- Re-exporting `model` derive macro (only needed in model files, not handlers)

## Technical Notes

### New Module: `autumn/src/prelude.rs`

```rust
//! Convenience re-exports for Autumn applications.
//!
//! Import everything commonly needed with a single glob:
//!
//! ```rust
//! use autumn::prelude::*;
//! ```
//!
//! This brings route macros, rendering types, extractors, and error
//! types into scope. For less common types (config, middleware,
//! upstream crate access), use targeted imports from `autumn::config`,
//! `autumn::middleware`, or `autumn::reexports`.

// ── Route macros ─────────────────────────────────────────────────
/// HTTP method route macros and collection macro.
pub use autumn_macros::{delete, get, main, post, put, routes};

// ── Rendering ────────────────────────────────────────────────────
/// Maud HTML templating types.
pub use maud::{html, Markup, PreEscaped};

// ── Extractors ───────────────────────────────────────────────────
/// Database connection extractor.
pub use crate::db::Db;
/// JSON request/response type.
pub use axum::Json;
/// Form data extractor.
pub use axum::extract::Form;

// ── Error handling ───────────────────────────────────────────────
/// Framework error and result types.
pub use crate::error::{AutumnError, AutumnResult};

// ── Application state ───────────────────────────────────────────
/// Shared application state (for custom extractors).
pub use crate::AppState;
```

### Module Declaration in `lib.rs`

Add to `lib.rs`:

```rust
pub mod prelude;
```

### What's NOT in the Prelude

Intentionally excluded to keep the prelude focused:

| Type | Why excluded |
|------|-------------|
| `AutumnConfig`, `ServerConfig`, etc. | Config internals — only needed in advanced customization |
| `Route`, `RouteMethod` | Framework internals — users don't construct routes manually |
| `RequestIdLayer`, `RequestId` | Middleware internals |
| `model` derive macro | Only used on model structs, not in handler files |
| `HTMX_VERSION` | Diagnostic constant, rarely imported |
| `app()` builder | Called once in `main()`, easy to import individually |
| `reexports::*` | Explicit upstream access, shouldn't be glob-imported |

### Backward Compatibility

The prelude is *additive* — it doesn't remove existing root-level exports. Users can still write `use autumn::get;` or `use autumn::Json;`. The prelude is an opt-in convenience, not a required migration.

### Compile Test

Add a doc test or integration test that proves the prelude works:

```rust

fn prelude_compiles() {
    use autumn::prelude::*;

    // Verify key types are in scope by using them in type position
    fn _handler(_db: Db) -> AutumnResult<Markup> {
        Ok(html! { "test" })
    }

    // Verify Json is in scope

        Json("ok")
    }
}
```

## Additional Notes

- Rust convention: preludes should contain types that are needed in "almost every file." Autumn's prelude targets *handler files* specifically — the most common file type in an Autumn project.
- The `model` macro is intentionally excluded because model files have different import needs (diesel types, schema, etc.). A separate `use autumn::model_prelude::*;` could exist in the future but is over-engineering for v0.1.
- If Sprint 8 adds new public types (e.g., from logging), evaluate whether they belong in the prelude. Logging types (`tracing::info!`, etc.) should NOT be in the prelude — users import `tracing` directly for custom instrumentation.
