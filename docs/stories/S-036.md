## User Story

As a **developer who just ran `autumn new`**
I want **`autumn setup` to download the correct Tailwind CLI binary for my platform**
So that **`cargo build` works immediately without me hunting down external tools**

---

## Description

### Background

The generated project's `build.rs` expects a Tailwind CLI binary at `target/autumn/tailwindcss` or on PATH. Without it, `cargo build` panics with a helpful error pointing to `autumn setup`. This story implements that command — it detects the host platform, downloads the correct Tailwind standalone CLI binary, verifies it with checksums, and places it where `build.rs` expects it.

Per NFR-005 (Build Determinism), `cargo build` itself never accesses the network. Tool downloads are an explicit user action via `autumn setup`.

### Scope

**In scope:**
- Platform detection: Linux x64/arm64, macOS x64/arm64, Windows x64
- Download Tailwind v4 standalone CLI binary from GitHub releases
- SHA256 checksum verification after download
- Place binary at `target/autumn/tailwindcss` (or `tailwindcss.exe` on Windows)
- Set executable permissions on Unix
- Progress output during download
- Re-download/update if `--force` flag is passed
- Skip download if binary already exists and is valid
- Clear error messages for network failures, checksum mismatches, unsupported platforms

**Out of scope:**
- Downloading other tools (future: Diesel CLI)
- Version pinning in `autumn.toml` (future enhancement)
- Offline mode / bundled binaries
- Proxy configuration (use system env vars)

## Technical Notes

### Dependencies to Add

`autumn-cli/Cargo.toml` needs HTTP client and crypto:
```toml
[dependencies]
clap = { version = "4", features = ["derive"] }
thiserror = "2"
reqwest = { version = "0.12", features = ["blocking", "rustls-tls"], default-features = false }
sha2 = "0.10"
hex = "0.4"
```

Using `reqwest` blocking (not async) because the CLI is synchronous. `rustls-tls` avoids OpenSSL dependency for cross-platform builds. `sha2` + `hex` for checksum verification.

### Platform Detection

```rust
fn detect_platform() -> Result<&'static str, SetupError> {
    match (std::env::consts::OS, std::env::consts::ARCH) {
        ("linux", "x86_64") => Ok("tailwindcss-linux-x64"),
        ("linux", "aarch64") => Ok("tailwindcss-linux-arm64"),
        ("macos", "x86_64") => Ok("tailwindcss-macos-x64"),
        ("macos", "aarch64") => Ok("tailwindcss-macos-arm64"),
        ("windows", "x86_64") => Ok("tailwindcss-windows-x64.exe"),
        (os, arch) => Err(SetupError::UnsupportedPlatform(os.into(), arch.into())),
    }
}
```

### Download URL

Tailwind v4 standalone CLI releases:
`https://github.com/tailwindlabs/tailwindcss/releases/download/v{version}/{binary_name}`

Pin a specific version (e.g., `v4.1.0`) as a const in the source. The checksum file is available at the same URL with `.sha256` suffix, or checksums can be hardcoded for the pinned version.

### Checksum Strategy

Option A (recommended): Hardcode SHA256 checksums for the pinned version in the source. This is deterministic and doesn't require a second HTTP request.

Option B: Download the `.sha256` file from GitHub and verify against it. More flexible but adds a network request.

Go with Option A — hardcoded checksums for the pinned Tailwind version. Update them when bumping the Tailwind version.

### Output Location

`target/autumn/tailwindcss` — matches what `build.rs` in generated projects looks for. The `target/` directory is already gitignored, so the binary won't accidentally be committed.

### Testing

- **Unit tests:** Platform detection (mock `consts::OS`/`consts::ARCH` by parameterizing the function)
- **Integration test:** Download to tempdir, verify checksum, verify file exists and is executable
- **Error tests:** Invalid platform, checksum mismatch (provide wrong expected hash)
- **Note:** Network-dependent tests should be marked `#[ignore]` for CI and run manually or in a separate CI job

## Additional Notes

- The Tailwind standalone CLI is ~50MB. Downloads may be slow on poor connections. Consider showing download speed/progress.
- On Windows, the binary needs the `.exe` extension. The `build.rs` template's `which()` function already checks for `.exe` variants.
- Future enhancement: `autumn setup --version 4.2.0` to pin a specific Tailwind version. Not needed for v0.1.
- The hardcoded checksums will need updating when Tailwind releases new versions. This is acceptable for v0.1 — the CLI ships with one known-good Tailwind version.
