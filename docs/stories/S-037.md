## User Story

As a **framework maintainer**
I want **an automated test that scaffolds a project, builds it, starts it, and verifies HTTP responses**
So that **template drift between `autumn-cli` and the `autumn` crate is caught before release**

---

## Description

### Background

Architecture decision TD-006 notes that `autumn-cli` has no runtime dependency on `autumn` — it generates projects from templates. The risk is template drift: the CLI generates code that references APIs that have changed or no longer exist. This E2E test is the mitigation.

The test exercises the full developer flow: `autumn new test-app` → `cd test-app` → `cargo build` → `cargo run` → HTTP request → verify response. If this test passes, a new user's first experience works.

This is the critical integration test identified in the sprint plan: "S-037 is the critical integration test: `autumn new test-app && cd test-app && cargo run` must work."

### Scope

**In scope:**
- Test that scaffolds a project into a temp directory
- Override `autumn` dependency to use local path (not crates.io)
- `cargo build` succeeds on the generated project
- `cargo run` starts a server (with timeout)
- HTTP GET to `localhost:3000/` returns 200 with "Welcome to Autumn!"
- HTTP GET to `localhost:3000/hello/world` returns "Hello, world!"
- HTTP GET to `localhost:3000/health` returns 200 with JSON
- Server shutdown after test

**Out of scope:**
- Database-dependent tests (no Postgres required)
- Tailwind CSS compilation (build.rs skipped or Tailwind not required for this test)
- Testing on all 3 OSes (CI handles cross-platform; this test runs on each)
- Performance benchmarking

### Sprint Notes

The generated project's `build.rs` tries to run Tailwind CLI, which won't be present in the test environment. The E2E test needs to either: (a) remove `build.rs` from the generated project before building, (b) provide a mock Tailwind binary, or (c) conditionally skip the Tailwind step. Option (a) is simplest — delete `build.rs` after scaffolding, since we're testing Autumn's runtime, not Tailwind integration.

## Technical Notes

### Test Location

`autumn-cli/tests/e2e.rs` — integration test in the CLI crate.

### Path Dependency Override

The generated `Cargo.toml` has `autumn = "0.1.0"` which won't resolve locally. Patch it after generation:

```rust
// Append path override to generated Cargo.toml
let cargo_toml_path = project_dir.join("Cargo.toml");
let mut content = fs::read_to_string(&cargo_toml_path)?;
content.push_str(&format!(
    "\n[patch.crates-io]\nautumn = {{ path = \"{}\" }}\n",
    autumn_crate_path.display()
));
fs::write(&cargo_toml_path, content)?;
```

The `autumn_crate_path` is resolved relative to the workspace root (e.g., `../../autumn` from the temp dir).

### Build.rs Handling

Remove `build.rs` from the generated project before building — Tailwind CSS compilation is not the focus of this test:

```rust
fs::remove_file(project_dir.join("build.rs"))?;
```

### Server Lifecycle

1. `cargo build` in generated project directory
2. Spawn `cargo run` as a child process
3. Poll `localhost:{port}` with retries (server may take a moment to start)
4. Make HTTP assertions
5. Kill the child process
6. Clean up temp directory

Use a random port (or the default 3000) and ensure cleanup even on test failure (use RAII guard or `Drop` impl).

### HTTP Client

Use `reqwest::blocking` (already a dependency from S-036) for HTTP assertions in the test.

### Timeout Strategy

```rust
// Build timeout: 120s (first compile is slow)
let build = Command::new("cargo")
    .args(["build"])
    .current_dir(&project_dir)
    .timeout(Duration::from_secs(120))
    .output()?;

// Server startup: poll with 10s timeout
for _ in 0..20 {
    if reqwest::blocking::get(format!("http://localhost:{port}/health")).is_ok() {
        break;
    }
    thread::sleep(Duration::from_millis(500));
}
```

### CI Considerations

- This test is slow (compiles a fresh project). Mark with `#[ignore]` or put in a separate test binary that CI runs explicitly.
- On Windows, `cargo run` spawns a process that needs `taskkill` to terminate (no SIGTERM). Use `child.kill()` which works cross-platform.
- The workspace root path must be resolved at test time, not hardcoded.

## Story Points Breakdown

- **Scaffold + path dependency patching:** 0.5 pts
- **Build step with timeout:** 0.5 pts
- **Server spawn + poll for readiness:** 0.75 pts
- **HTTP assertions (3 endpoints):** 0.5 pts
- **Cleanup and process management:** 0.5 pts
- **CI compatibility (Windows process kill):** 0.25 pts
- **Total:** 3 pts

**Rationale:** The individual pieces are straightforward, but orchestrating an external process (spawn, wait, verify, kill) across platforms is fiddly. The test is more "systems integration" than "business logic" — the complexity is in reliability, not logic.

## Additional Notes

- This is the most important test for confidence in the release pipeline. If this fails, a user's first experience is broken.
- Consider running this test nightly or on PRs that touch `autumn-cli/` or `autumn/src/` (path-based CI filtering).
- The build step will be slow (~30-60s first compile). This is expected and acceptable for an E2E test.
- Future: once `autumn` is on crates.io, add a variant that tests against the published crate (no path override).
