## User Story

As a **developer learning Autumn**
I want **a step-by-step tutorial that builds a complete todo application from scratch**
So that **I understand how all the framework pieces fit together and can confidently build my own apps**

---

## Description

### Background

The getting started guide (S-039) is a quick tour — it shows each feature briefly but doesn't build a cohesive application. The tutorial is the deep dive. It walks the reader through building the same todo app that exists as `examples/todo-app/` (S-041), but incrementally — explaining each decision, each pattern, each "why."

This is the largest documentation story in the project (8pts). It spans two sprints: Sprint 11 starts the outline and first chapters (1pt), Sprint 12 completes the full tutorial (7pts). This split lets S-042 (API docs) and S-044 (feature flags) get done in Sprint 11 while the tutorial outline solidifies.

### Scope

**In scope:**
- `docs/guide/tutorial.md` (or `docs/guide/tutorial/` directory with chapters)
- Complete walkthrough: empty directory → working todo app with all features
- Chapters covering: project setup, routes, database, models, templates, forms, htmx interactivity, JSON API, error handling, configuration, testing
- Code listings that match the `examples/todo-app/` reference implementation
- Explanations of Autumn conventions and patterns
- "Why" explanations for design decisions (why Maud over Tera, why Diesel over SQLx, etc.)
- Exercises/challenges at the end of chapters (optional stretch)

**Out of scope:**
- Deployment tutorial
- Authentication/authorization
- Advanced topics (custom middleware, extending the framework)
- Video content
- Interactive playground

### Tutorial Structure

```
docs/guide/tutorial.md (or directory)

Chapter 1: Project Setup
  - autumn new, project structure, cargo run, first response

Chapter 2: Routes and Handlers
  - #[get], #[post], path parameters, routes![]
  - How route macros generate code (brief, not overwhelming)

Chapter 3: Database Setup
  - docker-compose, diesel CLI, migrations
  - Creating the todos table

Chapter 4: Models and Queries
  - #[model] macro, Diesel schema, CRUD operations
  - The Db extractor — how it works

Chapter 5: HTML Templates with Maud
  - Maud syntax basics (for non-Maud users)
  - Layout pattern, todo list template, form template

Chapter 6: Styling with Tailwind CSS
  - How the build.rs pipeline works
  - Adding classes, responsive design basics

Chapter 7: Interactivity with htmx
  - What htmx does and why Autumn chose it
  - hx-post for form submission without page reload
  - hx-delete for inline delete
  - hx-swap and partial responses

Chapter 8: JSON API
  - GET /api/todos, POST /api/todos
  - Json<T> extractor and response type
  - Sharing logic between HTML and JSON handlers

Chapter 9: Error Handling
  - AutumnResult<T> and the ? operator
  - Custom status codes (.not_found(), .bad_request())
  - Validation errors

Chapter 10: Configuration and Production Defaults
  - autumn.toml sections
  - Environment variable overrides
  - Structured logging
  - Health check endpoint
  - Graceful shutdown

Chapter 11: What's Next
  - Links to API docs, example app, community
  - Ideas for extending the app (auth, categories, search)
```

### Relationship to S-041 (Example App)

The tutorial builds the SAME app as `examples/todo-app/`. At the end, the reader's code should be functionally identical to the example. This means:
- The tutorial is the narrative; the example is the reference implementation
- If the example changes, the tutorial must be updated (and vice versa)
- The tutorial can omit some polish that the example includes (e.g., the example may have more CSS refinement)

## Technical Notes

### Sprint Split Strategy

**Sprint 11 (1pt):**
- Create `docs/guide/tutorial.md` (or directory structure)
- Write the outline with chapter titles, goals, and key topics
- Write Chapter 1 (Project Setup) as the anchor — establishes tone and pace
- Stub remaining chapters with section headers and brief notes

**Sprint 12 (7pts):**
- Write Chapters 2-11
- Test all code examples end-to-end
- Cross-reference with `examples/todo-app/`
- Polish and review

### File Organization Options

**Option A: Single file** — `docs/guide/tutorial.md`
- Pros: Simple, one URL to share, easy to search
- Cons: Very long (~5000-8000 words)

**Option B: Directory** — `docs/guide/tutorial/`
- `index.md` — overview + table of contents
- `01-project-setup.md`
- `02-routes.md`
- ...
- Pros: Each chapter is focused, easier to maintain
- Cons: More files, need cross-chapter links

Recommendation: **Option B** — the tutorial is long enough to warrant chapters. Each chapter should be independently readable after the reader has completed previous ones.

### Code Listing Style

Show diffs where possible — don't repeat the entire file for a one-line change:

```markdown
Add the route handler to `src/routes/todos.rs`:

\```rust
#[get("/todos")]
async fn list(db: Db) -> AutumnResult<Markup> {
    // ...
}
\```

And register it in `src/main.rs`:

\```diff
- .routes(routes![index])
+ .routes(routes![index, list])
\```
```

### Testing Code Examples

Create a `docs/guide/tutorial-test/` directory (gitignored) where the tutorial can be followed step by step to verify all code works. This isn't checked into git — it's a QA step during writing.

### Tone Guide

- Second person ("You'll create...", "Add this route...")
- Present tense for instructions
- Brief asides for context ("Maud uses a Rust DSL instead of a template language — this means you get type-checking and IDE support for free")
- No filler paragraphs — every sentence teaches something or instructs an action
- Celebrate milestones ("Run `cargo run` — you should see your first todo in the list! 🎉" — okay, one emoji per milestone)

## Additional Notes

- The tutorial should NOT be a copy of the getting started guide with more detail. The guide is "here's what Autumn can do" (breadth). The tutorial is "let's build something real" (depth).
- Assume the reader has read the getting started guide OR is comfortable skipping Chapter 1.
- The htmx chapter (7) is the most likely to feel "magical" to readers — take extra care to explain what's happening under the hood (htmx sends AJAX, server returns HTML fragment, htmx swaps it into the DOM).
- Consider: should each chapter end with a "checkpoint" showing the full file state? Helps readers who get lost. May be verbose. Try it for Chapter 1 and see how it feels.
- The Diesel chapter should mention `diesel print-schema` and the `schema.rs` file — this is often confusing for Diesel newcomers.
