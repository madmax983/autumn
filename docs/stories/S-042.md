## User Story

So that **I can look up any type, function, or macro and immediately understand how to use it**

---

## Description

### Background

`cargo doc` is the standard Rust documentation tool and is often the first place developers look for API details. Autumn already has rustdoc comments on most public items, but they need to be reviewed for completeness, consistency, and working examples.

This story is not about writing a guide or tutorial — it's about ensuring that `cargo doc --open` produces a professional, complete API reference. Every public struct, enum, trait, function, and macro should have:
1. A one-line summary
2. A description of what it does and when to use it
3. At least one code example (compilable where possible)
4. Links to related items

### Scope

**In scope:**
- Review and enhance doc comments on all public items in the `autumn` crate
- Review and enhance doc comments on all proc macros in `autumn-macros`
- Crate-level documentation (`//!` in lib.rs) with overview and quick example
- Module-level documentation for each public module
- `# Examples` sections with compilable doc-tests where feasible
- `# Panics`, `# Errors` sections where applicable
- Cross-references between related types (`[`AutumnError`]`, etc.)
- Verify `cargo doc --no-deps` builds clean (no warnings)
- Verify doc-tests pass (`cargo test --doc`)

**Out of scope:**
- External documentation site (mdBook, etc.) — that's post-v0.1
- Getting started guide (S-039) or tutorial (S-040)
- Documentation for internal/private items
- Changelogs or release notes

## Technical Notes

### Doc-Test Patterns

**Route macros (proc macros can't easily have doc-tests):**
Use `# fn main() {}` wrapper or `no_run` / `ignore` attributes where async context is needed:
```rust
/// # Example
///
/// ```rust,no_run
/// use autumn_web::prelude::*;
///
/// #[get("/")]
/// async fn index() -> &'static str {
///     "Hello!"
/// }
/// ```
```

**Config struct (testable):**
```rust
/// # Example
///
/// ```rust
/// use autumn_web::config::AutumnConfig;
///
/// let config = AutumnConfig::default();
/// assert_eq!(config.server.port, 3000);
/// ```
```

**Error types (testable):**
```rust
/// # Example
///
/// ```rust
/// use autumn_web::error::AutumnError;
/// use http::StatusCode;
///
/// let err = AutumnError::bad_request(std::io::Error::other("oops"));
/// assert_eq!(err.status(), StatusCode::BAD_REQUEST);
/// ```
```

**Db extractor (no_run — needs async runtime + database):**
```rust
/// # Example
///
/// ```rust,no_run
/// use autumn_web::prelude::*;
/// use autumn_web::db::Db;
///
/// #[get("/users")]
/// async fn list(db: Db) -> AutumnResult<String> {
///     // Use db to query...
///     Ok("users".to_string())
/// }
/// ```
```

### Doc Warnings

Run `RUSTDOCFLAGS="-D warnings" cargo doc --no-deps` to catch:
- Missing doc comments on public items
- Broken intra-doc links
- Dead code in doc examples

### Module Organization for Docs

Each module's `//!` doc should follow this pattern:
```rust
//! # Module Name
//!
//! Brief description of what this module provides.
//!
//! ## Overview
//!
//! Longer explanation of the module's role in the framework.
//!
//! ## Example
//!
//! ```rust,no_run
//! // Quick usage example
//! ```
```

### Proc Macro Docs

Proc macros in `autumn-macros` are re-exported by `autumn`. Document them at the re-export site in `autumn/src/lib.rs` using `#[doc = "..."]` or rely on the proc macro's own docs propagating through the re-export. Verify which approach renders correctly in `cargo doc`.

### Priority Order

If time-constrained, document in this priority:
1. Crate-level overview + prelude module
2. Proc macros (get, post, routes, main, model) — most visible to users
3. Error types (AutumnError, AutumnResult) — most frequently encountered
4. Config structs — needed for any customization
5. Db extractor — needed for database work
6. Everything else (middleware, health, logging, route)

**Rationale:** Writing good documentation is surprisingly time-consuming. Each public item needs a summary, description, and working example. The proc macros are the hardest to document because their doc-tests need careful setup (async context, correct imports). The inventory is ~30 items, but many are related (config structs share patterns, extractors are re-exports needing only brief docs).

## Additional Notes

- Don't over-document re-exports. For `Form`, `Json`, `Path`, `Query` — a one-liner saying "Re-exported from Axum for convenience. See [axum::extract::Form] for full docs." is fine.
- For Maud re-exports (`Markup`, `html!`, `PreEscaped`), similarly link to upstream docs but add an Autumn-specific usage example.
- The `reexports` module exists so users can access `autumn_web::reexports::axum::Router` etc. Document *why* — "For advanced use cases where you need to access the underlying framework types directly."
- Consider adding `#![doc = include_str!("../../README.md")]` to `lib.rs` if it improves the crates.io landing page. Test this before committing.
