## User Story

As a **developer who only needs a JSON API (no HTML rendering)**
I want **to opt out of Maud, Tailwind, and htmx dependencies via Cargo feature flags**
So that **my compile times are faster and my binary is smaller when I don't need the full-stack features**

---

## Description

### Background

Autumn currently compiles all features unconditionally: Maud (HTML templating), htmx (embedded JS), Tailwind (build.rs integration), and the database layer. A developer building a JSON-only API doesn't need any rendering dependencies. A developer using a different CSS framework doesn't need Tailwind.

Cargo feature flags let users opt in to what they need. The default feature set should include everything (convention-over-configuration), but users can disable defaults for leaner builds.

Currently, the `autumn` crate's `Cargo.toml` has zero feature flags defined. All dependencies (maud, diesel, diesel-async, tower-http, etc.) are unconditional.

### Scope

**In scope:**
- Define feature flags in `autumn/Cargo.toml`
- Gate optional dependencies behind features
- Gate code paths with `#[cfg(feature = "...")]`
- Update `autumn::prelude` to conditionally export
- Ensure `default = [all features]` so existing users aren't broken
- Update the `autumn-macros` crate if any macros need feature awareness
- Document features in crate-level docs

**Out of scope:**
- Feature flags on `autumn-cli` (CLI always generates full-stack projects)
- Feature flags on `autumn-macros` (macros should work regardless)
- Per-example feature customization
- Runtime feature detection

### Feature Flag Design

```toml
[features]
default = ["maud", "htmx", "tailwind", "db"]

# HTML templating with Maud
maud = ["dep:maud"]

# Embedded htmx.js serving
htmx = []  # htmx.js is include_bytes!, feature gates the serving route

# Tailwind CSS build pipeline support
tailwind = []  # Gates build.rs template, no runtime deps

# Database layer (Diesel + async pool)
db = ["dep:diesel", "dep:diesel-async", "dep:libsqlite3-sys"]
```

## Technical Notes

### Cargo.toml Changes

```toml
[features]
default = ["maud", "htmx", "tailwind", "db"]
maud = ["dep:maud"]
htmx = []
tailwind = []
db = ["dep:diesel", "dep:diesel-async", "dep:libsqlite3-sys"]

[dependencies]
# Always required
autumn-macros = { path = "../autumn-macros" }
axum = { workspace = true }
tokio = { workspace = true }
serde = { workspace = true }
serde_json = "1.0"
thiserror = { workspace = true }
tower = { workspace = true }
tower-http = { workspace = true }
tracing = { workspace = true }
tracing-subscriber = { workspace = true }
toml = { workspace = true }
http = { workspace = true }
uuid = { version = "1.0", features = ["v4"] }
pin-project-lite = "0.2"

# Optional
maud = { workspace = true, optional = true }
diesel = { workspace = true, optional = true }
diesel-async = { workspace = true, optional = true }
libsqlite3-sys = { workspace = true, optional = true }
```

### Code Gating Patterns

**Module-level gating (lib.rs):**
```rust
#[cfg(feature = "db")]
pub mod db;

#[cfg(feature = "maud")]
pub use maud::{Markup, PreEscaped, html};
```

**Conditional prelude exports (prelude.rs):**
```rust
#[cfg(feature = "db")]
pub use crate::db::Db;

#[cfg(feature = "maud")]
pub use maud::{Markup, PreEscaped, html};
```

**Conditional htmx route (app.rs or wherever routes are mounted):**
```rust
#[cfg(feature = "htmx")]
{
    router = router.route("/static/js/htmx.min.js", get(htmx_handler));
}
```

**Health handler — conditional pool status:**
```rust
pub async fn handler(State(state): State<AppState>) -> impl IntoResponse {
    let mut status = json!({ "status": "ok" });

    #[cfg(feature = "db")]
    if let Some(pool) = &state.pool {
        status["pool"] = json!({
            "size": pool.status().size,
            "available": pool.status().available,
        });
    }

    Json(status)
}
```

**AppState — conditional pool field:**
```rust
#[derive(Clone)]
pub struct AppState {
    #[cfg(feature = "db")]
    pub pool: Option<Pool<AsyncPgConnection>>,
}
```

### Macro Considerations

The `#[model]` macro (in `autumn-macros`) generates Diesel derives. If `db` feature is off, the macro still exists but the generated code won't compile because Diesel traits aren't available. Options:

1. **Document it:** "`#[model]` requires the `db` feature." (simplest)
2. **Compile error in macro:** Check for `db` feature at macro expansion time (complex, may not be feasible from proc-macro crate)
3. **Gate the re-export:** Don't re-export `model` from `autumn` when `db` is off

Recommendation: Option 3 — gate the `model` macro re-export behind `#[cfg(feature = "db")]`.

### CI Impact

The CI pipeline (S-043) should test feature combinations:
```yaml
- cargo test -p autumn --no-default-features
- cargo test -p autumn --features maud
- cargo test -p autumn --features db
- cargo test -p autumn  # all defaults
```

This is a note for CI, not a requirement for this story.

## Additional Notes

- The `tailwind` feature flag is unusual — it has no runtime dependencies. It exists so that documentation and tooling can check if Tailwind is expected. The `build.rs` template that `autumn new` generates already handles Tailwind independently.
- Don't gate `tower-http` behind features — static file serving (`ServeDir`) is useful even without Maud/Tailwind.
- The `extract` module re-exports (`Form`, `Json`, `Path`, `Query`) should always be available — they're from Axum, not optional deps.
- Consider: should `Json` response type require `serde_json`? Currently `serde_json` is always included. Keep it unconditional — JSON is fundamental to web APIs.
